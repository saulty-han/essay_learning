ReAcTree: Hierarchical LLM Agent Trees

带有控制流的层次化 LLM 智能体树 — 让长周期任务规划从"单线冒险"变为"团队协作"

📄 AAMAS 2026 Full Paper  |  arXiv: 2511.02424  |  GitHub: Choi-JaeWoo/ReAcTree

AAMAS 2026 Hierarchical Planning Control Flow Embodied Agent Dual Memory

★ 整体架构总览

ReAcTree 的核心思想是将复杂的长周期任务分解为一棵动态构建的智能体树（Agent Tree），其中每个节点是一个 LLM 智能体，负责完成一个子目标。不同于 ReAct 的单一轨迹，ReAcTree 通过控制流节点（顺序、循环、条件分支）协调子智能体的执行策略，并利用双记忆系统（情景记忆+工作记忆）实现跨子目标的知识共享。

▲ ReAcTree 架构总览：根智能体分解目标→控制流节点协调→子智能体执行→双记忆支持

🌳 动态智能体树

不同于静态的行动树，ReAcTree 在子目标空间中动态构建树结构。每个子智能体可以进一步扩展为子树，实现递归式的任务分解。

🔀 控制流节点

引入编程语言式的控制流（顺序、循环、条件、并行），使智能体能够表达复杂的执行策略，而非简单的线性推理链。

🧠 双记忆系统

情景记忆提供子目标级的历史示例参考，工作记忆在节点间共享环境观察，避免信息丢失和冗余交互。

0 核心问题与研究动机

具身智能体在长周期任务上面临一个核心困境：单体轨迹纠缠。以 ReAct 为代表的方法，将所有历史决策和观察压入一条线性轨迹——随着任务步数增加，上下文窗口被冗余信息填满，模型难以区分关键信息和噪声，导致"遗忘"早期决策、"迷路"在无关观察中。就像一个人同时处理十个子任务，把所有笔记混在一张纸上，越写越混乱。

🔑 核心问题

如何让 LLM 智能体在长周期任务中保持结构化的推理能力？具体来说，如何将复杂目标分解为相对独立的子目标，使每个子目标可以在精简的上下文中被高效解决，同时保持子目标之间的协调和信息流通？

ReAcTree 的回答是：用树替代线性轨迹，用控制流替代硬编码策略，用双记忆替代单一上下文。树结构天然支持任务的层次分解；控制流节点提供了灵活的执行策略编排；双记忆系统在保持信息流通的同时避免了上下文膨胀。

1 背景知识与前置概念

ReAct 及其局限性

ReAct 是智能体推理的奠基性工作，它让 LLM 交替生成"思考"和"行动"步骤。但 ReAct 的线性轨迹设计在长周期任务中暴露了三个问题：(1) 上下文膨胀——所有历史步骤累积在一条轨迹中，超出窗口限制后早期信息丢失；(2) 纠缠决策——不同子任务的决策交织在一起，模型难以追踪每个子任务的进展；(3) 错误传播——一步错误会影响后续所有步骤，没有隔离机制。

行为树 (Behavior Tree) 与智能体推理

行为树是机器人学和游戏 AI 中广泛使用的任务编排结构，通过控制流节点（选择器、序列、并行等）组织叶节点（具体动作）。ReAcTree 借鉴了行为树的思想，但做了关键改造：(1) 叶节点不是预定义的动作，而是可递归扩展的 LLM 智能体；(2) 树是动态构建的，而非预先定义的；(3) 控制流节点由 LLM 自动选择和参数化，而非人工设计。

情景记忆 vs. 工作记忆

这是认知心理学中的经典区分，ReAcTree 将其引入智能体推理。情景记忆（Episodic Memory）存储历史交互的具体案例——"上次遇到类似情况时我是怎么做的"，通过相似度检索来指导当前决策。工作记忆（Working Memory）存储当前任务的关键环境观察——"房间里有红色方块"，在树的节点间共享以避免重复探索。前者提供"经验参考"，后者提供"当前状态"。

2 方法论：ReAcTree 核心模块

2.1 动态智能体树构建

ReAcTree 的树构建过程是自顶向下、按需扩展的：根智能体接收总目标后，首先将其分解为若干子目标，每个子目标生成一个子智能体节点。子智能体执行时可以进一步分解自己的子目标，递归构建子树。关键设计：树在子目标空间而非动作空间中构建——这意味着每个节点关注的是"要完成什么"而非"怎么做"，使得高层规划与低层执行解耦。

2.2 控制流节点

控制流节点是 ReAcTree 区别于简单递归分解的关键创新。四种控制流节点：

• 顺序节点（Sequence）：子目标按顺序依次执行，前一个完成后才开始下一个。适用于有明确依赖关系的子目标。

• 循环节点（Loop）：重复执行子目标直到满足终止条件。适用于需要反复尝试或迭代改进的场景（如"反复搜索直到找到目标"）。

• 条件节点（If/Else）：根据环境观察决定执行哪个分支。适用于需要根据中间结果调整策略的场景。

• 并行节点（Parallel）：同时执行多个子目标。适用于独立子任务的并行加速。

2.3 双记忆系统

情景记忆：每个智能体节点在执行前，从外部情景记忆库中检索与当前子目标语义相似的历史案例。这些案例提供了"类似子目标之前是怎么完成的"参考，帮助智能体少走弯路。检索基于子目标描述的嵌入相似度。

工作记忆：所有智能体节点共享一个工作记忆区，存储环境的关键观察结果。当一个节点发现"房间有红色钥匙"时，这个信息会写入工作记忆，其他需要"红色钥匙"的节点可以直接读取，无需重复探索。这大大减少了冗余的环境交互。

3 核心创新点深度解析

创新1：子目标空间的动态树 vs. 动作空间的静态树

此前的方法（如 RAP、LATS）在动作空间构建搜索树——每个节点是 LLM 生成的一个具体动作。ReAcTree 在子目标空间构建树——每个节点是一个子目标及其对应的智能体。这个区别看似微小，实则深刻：动作空间树的分支因子巨大（每个状态可能有数十种可用动作），搜索效率低；子目标空间树的分支因子小（通常3-5个子目标），且每个子目标可以由专用智能体独立处理，互不干扰。

创新2：编程式控制流的引入

将编程语言的控制流概念引入智能体规划是 ReAcTree 的标志性贡献。传统方法要么是纯线性的（ReAct），要么是纯搜索的（ToT, LATS），缺乏表达"循环直到成功"或"如果A则做B否则做C"这类条件执行策略的能力。控制流节点让智能体能够表达策略性知识——不只是"做什么"，还包括"怎么组织做"。

创新3：双记忆的互补设计

情景记忆提供纵向经验复用（跨任务、跨时间），工作记忆提供横向信息共享（同任务、跨节点）。两者互补：情景记忆告诉智能体"以前怎么做的"，工作记忆告诉智能体"现在环境是什么样的"。没有情景记忆，智能体每次都要从零开始；没有工作记忆，子智能体之间无法协调，可能重复探索或互相矛盾。

5 关键技术细节

子目标分解的提示工程

ReAcTree 使用精心设计的提示模板让 LLM 执行子目标分解。提示包含：(1) 当前目标描述；(2) 当前环境观察；(3) 已完成的子目标列表；(4) 从情景记忆检索到的相似案例。LLM 需要输出：分解后的子目标列表、每个子目标对应的控制流类型（顺序/循环/条件/并行），以及控制流参数（如循环的终止条件）。

情景记忆的检索机制

情景记忆库存储历史交互中的"子目标-行动-结果"三元组。检索时，将当前子目标的语义描述编码为向量，与记忆库中的子目标描述计算余弦相似度，返回 top-k 最相似的案例。关键设计：检索粒度是子目标级而非任务级——这确保了检索到的案例与当前子步骤高度相关，而非只是与总目标相似但具体步骤不同。

工作记忆的共享与更新

工作记忆是一个键值对结构，存储环境中的关键事实（如"location: kitchen"、"holding: red_key"）。每当子智能体执行一个动作并观察环境变化时，它会将新的观察写入工作记忆。下一个子智能体开始执行前，会从工作记忆中读取所有相关事实。这种设计避免了子智能体之间的重复探索——如果一个节点已经发现"红色钥匙在桌子上"，其他需要红色钥匙的节点不需要再去搜索。

消融实验分析

论文进行了系统的消融实验，逐步移除 ReAcTree 的各组件：

• 去掉控制流节点（只保留树分解）：性能下降约 8-10%，说明控制流对策略表达至关重要

• 去掉情景记忆：性能下降约 5-7%，说明经验复用显著减少试错

• 去掉工作记忆：性能下降约 10-12%，说明跨节点信息共享对避免冗余交互最为关键

• 去掉树结构（退化为 ReAct）：性能下降约 30%，说明层次分解是最核心的贡献

7 设计精妙之处与局限性

✨ 设计精妙之处

1. 子目标空间树 vs. 动作空间树

在子目标空间建树，分支因子小、语义清晰，搜索效率远高于在动作空间建树。每个子智能体只需关注自己的子目标，上下文精简。

2. 控制流的编程隐喻

将编程语言的顺序/循环/条件/并行引入智能体规划，使策略表达从"只能线性"升级为"可编程"，大大提升了复杂执行策略的表达能力。

3. 双记忆的纵横互补

情景记忆（纵向跨任务经验）和工作记忆（横向跨节点状态）互补，既避免从零开始，又避免重复探索。

⚠️ 局限性

1. 仅在具身任务上验证

实验限于 WAH-NL 和 ALFRED 两个具身任务基准，未在代码生成、科学推理等其他长周期任务上验证泛化性。

2. 子目标分解质量依赖 LLM

分解质量完全依赖 LLM 的规划能力，弱模型可能产生不合理的子目标分解，导致树结构低效或错误。

3. 控制流的选择缺乏验证

LLM 自动选择控制流类型和参数，但缺乏对选择正确性的验证机制——如果选错了循环终止条件，可能导致无限循环。

8 Q&A 深度解析

Q1: ReAcTree 和 Tree of Thoughts (ToT) 的本质区别是什么？
ToT 在推理步骤级别建树，每个节点是一个中间推理状态，搜索目标是找到最佳推理路径。ReAcTree 在子目标级别建树，每个节点是一个可独立执行的智能体。区别在于：ToT 的树是"搜索"的产物——通过探索多条路径找到最优解；ReAcTree 的树是"分解"的产物——将大任务拆成小任务再组合执行。ToT 重在"搜索策略"，ReAcTree 重在"任务组织"。
Q2: 控制流节点在实际中如何被 LLM 生成？
LLM 在分解子目标时，同时输出每个子目标的控制流类型和参数。例如："子目标1：找到钥匙（Loop，终止条件=手中持有钥匙）→ 子目标2：开门（Sequence）"。提示模板中包含四种控制流的定义和示例，LLM 按格式输出。控制流的选择基于 LLM 对任务结构的理解——需要重复的用 Loop，有依赖的用 Sequence，需要判断的用 If/Else。
Q3: 工作记忆和情景记忆的存储开销如何？
工作记忆是键值对，通常只存储 10-30 个关键事实，开销很小。情景记忆库的大小取决于历史交互数量——论文中每个任务存储约 5-10 个关键子目标案例，使用向量检索，开销可控。主要成本在检索时的 embedding 计算，但这比额外的一次 LLM 调用便宜得多。
Q4: 为什么 WAH-NL 上提升比 ALFRED 更大？
WAH-NL 的任务更长、子目标更多（平均 15-20 步 vs. ALFRED 的 8-12 步），ReAct 在长轨迹上的上下文膨胀问题更严重。ReAcTree 通过树分解将长轨迹切分为短子轨迹，在 WAH-NL 上收益更大。这验证了核心假设：任务越长，ReAcTree 的优势越明显。
Q5: ReAcTree 可以和强化学习结合吗？
理论上可以，但论文没有探索。一种可能的方式是：用 RL 训练 LLM 学习更优的子目标分解策略和控制流选择。当前的分解完全依赖 LLM 的零样本规划能力，如果通过 RL 训练 LLM 在特定环境中最优地分解目标，可能进一步提升性能。这是自然的未来方向。
Q6: 并行节点在实际中如何实现？
在当前实现中，并行节点是逻辑上的并行而非物理并行——子智能体仍然串行执行，但它们的执行顺序不影响结果（因为子目标相互独立）。真正的物理并行需要多个 LLM 实例同时调用，会增加成本但减少延迟。论文选择了逻辑并行，这是一个务实的工程权衡。
Q7: 如果子目标分解错误怎么办？
这是 ReAcTree 的一个开放问题。当前没有显式的分解错误恢复机制——如果子目标分解不合理（如遗漏了关键步骤或创建了冗余子目标），整棵树的执行效率会下降。一种可能的改进是引入分解验证步骤：在执行前，让另一个 LLM 检查分解的合理性，或引入回溯机制允许重新分解。
Q8: ReAcTree 的核心启示对其他领域有什么启发？
核心启示是"分而治之+可控编排"的思想——将复杂任务分解为独立子任务，用结构化的控制流编排执行，用记忆系统维持上下文。这个思想不仅适用于具身智能体，也适用于代码生成（将大项目拆分为模块）、科学推理（将复杂实验拆分为子实验）、多跳问答（将长推理链拆分为子问题）等场景。

9 代码使用与实战

git clone https://github.com/Choi-JaeWoo/ReAcTree.git
cd ReAcTree

# 安装依赖
pip install -r requirements.txt

ReAcTree/
├── reactree.py          # 核心 ReAcTree 实现
├── agent_node.py       # 子智能体节点
├── control_flow.py     # 控制流节点实现

  --env wah_nl \
  --model qwen/Qwen2.5-72B-Instruct \

  --env wah_nl \
  --model qwen/Qwen2.5-72B-Instruct \
  --method reactree \
  --working_memory False
