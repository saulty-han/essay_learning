1 研究动机：树搜索实现的紧耦合之痛

现有树搜索实现（ToT / RAP / ReST-MCTS）多为任务特化，跨域 / 跨方法需大量重写。核心缺陷是算法与领域逻辑紧耦合，组件不可复用——用户想换方法或换任务都得重写整体。

算法与领域耦合

任务逻辑被塞进整体配置，每换一种搜索算法或领域都要重新实现，复用几乎为零。

本文主张

LiTS 把 LLM 树搜索分解为 Policy / Transition / RewardModel 三个可复用组件，通过装饰器注册机制实现"算法与领域的正交复用"，让用户"改所需、复用其余"。

核心创新

三组件分解 + 装饰器注册表，使同一组件可跨任务（环境规划 / 工具使用 / 语言推理）与跨算法（MCTS / BFS / ReAct）复用，并发现无限动作空间中 LLM 策略多样性是瓶颈。

2 框架概览：三层模块化结构

LiTS 分三层：lits.structures（Action → Step → State → Node 通用类型）、lits.components（Policy / Transition / RewardModel）、lits.agents（MCTS / BFS / ReAct 等）；lits.prompts 集中注册、lits.tools 兼容 LangChain。

端到端流程

CLI 按注册名解析组件 → Agent 编排组件执行搜索 / 链推理 → InferenceLogger 记录。BaseTreeSearch 处理外围逻辑，子类只写 search()，组件与算法正交，可跨任务 / 算法复用。

3 Policy 组件

Policy 从状态生成候选动作。框架内置多种实现：ConcatPolicy（拼接式）、ToolUsePolicy（工具使用式）、RAPPolicy（RAP 式），用户可注册自己的策略。

注册与复用

通过 @register_policy 装饰器注册，CLI 用 --policy 指定。同一 Policy 可复用于不同搜索算法（MCTS / BFS），实现"算法 × 领域"的正交组合。

4 Transition 组件

Transition 执行动作产出新状态，可调用工具。内置 BlocksWorldTransition、CrosswordsTransition 等，失败可确定性回退保证分支多样。

状态转移

Transition 把 (state, action) 映射为新 state，并负责工具调用与副作用。在有限动作空间（如 BlocksWorld）靠确定性回退保证分支多样性。

注册

@register_transition 装饰器 + 数据集加载器（同 key），CLI 切 --dataset 即可换域，无需改动算法代码。

5 RewardModel 组件

价值引导

RewardModel 评估动作质量引导树搜索，内置 GenerativePRM（生成式过程奖励）与 EnvGroundedPRM（环境接地奖励）。链方法（如 ReAct）无需 RewardModel。

瓶颈发现

在无限动作空间（如 Crosswords）下，LLM 策略多样性而非奖励质量是瓶颈——升温采样在 token 级而非动作级探索，导致模式崩溃（重复率 81.1%）；在有限动作空间（BlocksWorld）靠确定性回退保证分支多样。

6 三组件的可复用框架整合

装饰器注册表

@register_policy / @transition / @reward / @dataset / @search / @resource，CLI 用 --policy / --transition / --reward / --search_framework 组合。组件与算法正交，可跨任务 / 算法复用；InferenceLogger 提供可观测的检查点与日志。

8 敏感性分析：模式崩溃发现

无限 vs 有限动作空间

无限文本动作空间下 LLM 无校准随机策略头，升温 0.8→1.2 仍 81.1% 重复——说明瓶颈是策略多样性而非奖励质量；有限动作空间（BlocksWorld）靠确定性回退保证分支多样；工具使用任务奖励模型质量是瓶颈。

9 代码使用与实战（含 GitHub 源码、安装命令、使用示例）

官方仓库

代码仓库：https://github.com/xinzhel/lits-llm（RMIT University，Apache 2.0）。含安装说明与可运行示例；每运行写单一 save_dir/config.json + 检查点，可用 lits-eval --save_dir 复现。

# 1) 安装
pip install lits-llm

# 2) 组合组件运行 MCTS（Policy + Transition + Reward + 数据集）
lits run --search_framework mcts \
    --policy rap_policy --transition blocksworld \
    --reward generative_prm --dataset blocksworld

# 3) 复现实验结果
lits-eval --save_dir ./runs/blocksworld_mcts

# 4) 加新域：注册 Transition 子类 + 数据集加载器（同 key），切 --dataset
python -m lits.register --transition my_transition

复现要点：组件通过装饰器注册，CLI 组合即可；BaseTreeSearch 处理外围逻辑，子类只写 search()；InferenceLogger 提供检查点与日志，便于对比不同算法 / 领域组合。

10 总结与未来展望

LiTS 把 LLM 树搜索分解为 Policy / Transition / RewardModel 三组件，用装饰器注册表实现算法与领域的正交复用，在环境规划、工具使用、语言推理三类任务上验证有效，并发现无限动作空间中 LLM 策略多样性是瓶颈。

Q1：与 LLM Reasoners 区别？
LR 把任务逻辑塞进整体配置需每法重写；LiTS 拆为可复用组件，换算法 / 换域只需改对应组件。
Q2：如何加新域？
环境类只需注册 Transition 子类 + 数据集加载器（同 key），CLI 切 --dataset 即可，无需改算法。
Q3：模式崩溃含义？
无限文本动作空间下 LLM 无校准随机策略头，升温采样在 token 级而非动作级探索，导致重复率高。

局限与未来

仅演示级规模；模式崩溃结论基于单环境；工具需预先注册（计划接 MCP）；内置仅 MCTS / BFS；吞吐需并发批处理。
