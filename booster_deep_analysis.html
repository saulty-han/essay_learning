1 研究动机：调优器的适应性困境

数据库管理系统 (DBMS) 暴露了数万亿种可能的配置选项（旋钮、索引、查询提示），同时应用负载不断演变。 现有自动调优器虽然能针对固定工作负载找到优良配置，但在环境变化（负载漂移、跨 Schema 迁移）时却力不从心。

刚性与脆弱性 (Rigidity & Brittleness)

基于成本的调优器：缺乏将先验洞察注入搜索过程的机制——即便历史经验揭示了两个索引的不良交互，也无法优先排除它们。

基于 ML 的调优器：环境变化导致内部模型表示损坏（如 Q1 不再存在），必须重新优化，而预训练因目标环境差异和重新评估开销过大而不可行。

工作负载粒度问题 (Workload Granularity)

现有调优器在工作负载级别做适应性调整，使用负载级遥测来映射历史配置。但这种方式无法跨配置组合单个查询的洞察——例如 C1 优化了 Q1，C2 优化了 Q2，调优器无法将两者最佳方面组合起来。

LLM 调优器同样忽视了按查询逐个增强提示的机会。

关键实验证据 (Fig.1)：在 Transfer (TPC-H → DSB) 和 Drift (DSB 50% 参数变化) 场景下，"Total Knowledge Remix"（完全重组历史知识）比"Continue From History"（从历史继续调优）发现性能提升 13%–74%。这揭示了历史知识中蕴藏的巨大潜力。

2 框架概览：Booster 三阶段架构

Booster 作为现有调优器的"加速器"插件：分析历史制品仓库，将发现注入调优器，被注入的调优器进一步精炼配置并存回制品。

图：Booster 三阶段架构流程图 — Phase I (紫) → Phase II (橙) → Phase III (绿)

核心思想：Booster 不替代现有调优器，而是加速其适应环境变化。它通过查询级（而非工作负载级）的洞察重混历史知识，实现了"Total Knowledge Remix"。

3 Phase I：经验分析 — QConfig 构建与向量化

Booster 监控调优器的制品仓库，将非结构化的历史经验标准化为可搜索的 QConfig 对象，并向量化存入向量数据库。

🎯 这一阶段要解决的问题
调优器（如 Proto-X、UniTune）在过去 12 小时调优过程中，会在仓库里留下成千上万的"调优制品"——日志、配置文件、查询计划文本、性能数据。但这些制品有三个致命问题：
  • 格式不统一：每个调优器的日志格式不同（Proto-X 用二进制，UniTune 用 JSON，PGTune 用文本），无法跨调优器复用
  • 不可搜索：当一个新查询来了，无法快速找到"历史上这个查询长什么样、最优配置是什么"
  • 粒度太粗：传统做法只关心整体配置（"我整个工作负载用配置 C 跑了 50s"），但忽略了"配置 C 对 Q1 提升了 3 倍但对 Q5 反而变慢了"这种查询级洞察

Phase I 的目的就是把"原始日志垃圾箱"转化为"按查询语义可索引的经验库"，为后续 Phase II 的精准检索做好准备。

📦 QConfig 是什么？— 用 TPC-H Q1 实例讲清楚

QConfig（Query-Configuration object）是 Booster 设计的原子经验单元。它把"一个查询在某次调优中，被某个配置作用、产生了什么计划、跑了多久"封装成一个可序列化对象。可以理解为调优历史的"快照卡片"。下面以 TPC-H Q1（统计订单状态）为例展示一个真实的 QConfig： 

// ====== QConfig 实例：TPC-H Q1 的某次调优快照 ======
{
  "qconfig_id": "tpch_q1_proto_x_trial2_step47",
  "schema_metadata": {
    "tables": ["lineitem(6M rows, 16 cols)"],
    "existing_indexes": ["idx_lineitem_shipdate"],
    "stats": {"l_shipdate_ndv": 2526, "l_returnflag_ndv": 3}
  },
  "query": {
    "sql": "SELECT l_returnflag, l_linestatus, sum(l_quantity)...",
    "template": "SELECT l_returnflag, l_linestatus, sum(?), ... WHERE l_shipdate <= ?",
    "plan": "GroupAggregate -> Sort -> SeqScan(lineitem) [filter: l_shipdate<=?]"
  },
  "configuration": {
    "system_knobs": {"work_mem": "256MB", "max_parallel_workers_per_gather": 4},
    "indexes": ["idx_lineitem_shipdate_returnflag(l_shipdate, l_returnflag)"],
    "query_hints": ["SET enable_sort = OFF", "USE INDEX idx_..."]
  },
  "performance": {"runtime_ms": 1820, "plan_cost": 45230},
  "links": {
    "prev": "tpch_q1_proto_x_trial2_step46",  // 上一步配置（运行时 2410ms）
    "next": "tpch_q1_proto_x_trial2_step52"   // 下一步配置（运行时 1620ms，更优！）
  }
}
💡 为什么这样设计？ 每个 QConfig 既保留了"过去"（schema、查询、配置、计划），又通过 links 字段连接到"演进路径"（上一步/下一步配置）。这意味着 Phase II 检索到一个 QConfig 后，可以沿着 next 指针找到"它最终演化成了什么样"——找到的是 最优终态，而非中间废稿。

QConfig 数据结构

每个 QConfig 封装了一个查询在某配置下的行为，包含：

Schema 元数据

查询信息

🔍 Schematic 多视角向量化 — 为什么要 6 个向量？

一个查询可以从不同的"语义粒度"来描述。Schematic 就是这些不同粒度的文本表示。Booster 为每个 QConfig 生成 3 种类型 × 2 种变体 = 6 个 Schematic 文本，再分别嵌入为 6 个 Identity Vector (IV)。这种设计让检索能够在不同抽象层次上找到最相似的历史经验。 

// ====== TPC-H Q1 的 6 种 Schematic 表示 ======

[1] SQL 原文 (原始)
   "Schema: lineitem(6M rows). Query: SELECT l_returnflag, l_linestatus,
    sum(l_quantity), sum(l_extendedprice * (1 - l_discount)) FROM lineitem
    WHERE l_shipdate <= '1998-09-02' GROUP BY l_returnflag, l_linestatus"

[2] SQL 原文 (匿名化)  // 隐藏字面量、表名脱敏
   "Schema: TABLE_A(LARGE). Query: SELECT col_a, col_b, sum(col_c),
    sum(col_d * (1 - col_e)) FROM TABLE_A WHERE col_f <= ? GROUP BY col_a, col_b"

[3] 查询模板 (原始)  // 参数化但保留结构
   "Aggregation query on lineitem with date filter and group by 2 columns"

[4] 查询模板 (匿名化)
   "Aggregation query on TABLE_A with predicate filter and group by 2 attrs"

[5] 执行计划 (原始)  // 包含具体表/索引名
   "GroupAggregate -> Sort -> SeqScan(lineitem) [filter: l_shipdate<=?]"

[6] 执行计划 (匿名化)
   "GroupAggregate -> Sort -> SeqScan(TABLE_A) [filter: col_f<=?]"
💡 为什么需要"匿名化"变体？ 这是支持跨 Schema 迁移的关键设计！比如 TPC-H 的 lineitem 表和 DSB 的 store_sales 表，结构都是"大事实表 + 日期过滤 + GroupBy"。如果用原始 SQL 检索，"lineitem" ≠ "store_sales"，相似度低；但用匿名化后，两者都变成"TABLE_A WHERE col_f<=?"，相似度高——这样调优 TPC-H 学到的索引经验就能被迁移到 DSB 上。

Schematic 构建 — 多视角向量化

每个 QConfig 生成多种 Schematic（文本表示），涵盖从高层到低层的查询语义：

Schematic 类型Schema 部分查询部分变体主要用途
SQL 原文Schema 元数据SQL 文本原始 / 匿名化同 schema 精确匹配 / 跨 schema 迁移
查询模板Schema 元数据参数化 SQL原始 / 匿名化抗参数漂移（drift 场景核心）
执行计划Schema 元数据Plan 文本原始 / 匿名化结构相似 (Join/Scan 形态)

每种 Schematic 通过 Voyage 3 Large 嵌入模型生成一个 Identity Vector (IV)，共 6 个 IV（3 类型 × 2 变体）。

为什么需要链接？ 最语义相似的 QConfig 往往来自同一查询在初始配置下的记录——但它不包含任何有用的调优指导。通过链接追踪到最优下游 QConfig，才能获取真正有价值的配置参考。

把"每个调优器各说各话的乱日志"整理成"按查询语义可向量检索的经验库（QConfig + 6IV + Link）"。这是后两阶段能够"按查询精准检索经验"和"沿链接追溯最优"的基础设施。

4 Phase II：引导推荐 — 检索 + LLM 生成 + Sanitizer

当环境变化时，Booster 对工作负载中每个查询，从向量数据库检索相关历史经验，增强 LLM 提示词生成配置建议，再经 Sanitizer 清洗得到可行的查询 Seeds。

🎯 这一阶段要解决的问题
Phase I 已经把历史经验整理成可搜索的 QConfig 库了。现在面对一个新的工作负载（比如 schema 变了、查询参数漂移了），核心问题是：
  • 对每一个查询，我应该用什么配置？ —— 而不是"对整个工作负载用一个全局配置"
  • 历史中相似的查询用什么配置最有效？—— RAG 思想：retrieval-augmented generation
  • LLM 输出的配置可能不合法（无效索引、错误参数、不存在的列）—— 需要 Sanitize 清洗

Phase II 的目的就是为每个查询生成一组高质量的"候选配置"（Seeds）。注意：这里产出的还不是最终配置，而是"针对每个查询的局部建议"，等 Phase III 来组合成全局解。

Step 1: 查询向量化 + 相似检索

对目标查询生成 Schematic 和 IVs（与 Phase I 相同流程），然后在向量数据库中按欧氏距离检索同类 Schematic 的最相似 QConfig。关键设计：仅在同类型 Schematic 间比较（SQL 原文 vs SQL 原文，匿名化 vs 匿名化），避免跨类型误排。

为什么要在 6 种 Schematic 上都检索一遍？ 因为不同视角擅长捕获不同相似性——SQL 原文擅长"同库精确匹配"，匿名化模板擅长"跨库迁移"，执行计划擅长"形态相似"。把所有视角的 top-k 合并，能最大化召回率。

Step 2: 链接追踪 — 获取最优下游 QConfig

从每个检索到的 QConfig 出发，沿 next 链接追踪到最优下游 QConfig（运行时最短的那个）。这确保 LLM 参考的是历史中表现最好的配置，而非初始状态的"空"配置。

这步如果没有，会发生什么？ 检索器会优先返回最相似的——但最相似的往往是 step1（默认配置），因为它没被任何调优手段"扭曲"，文本/计划保持最原始的形态。如果直接把 step1 给 LLM，LLM 看到的是"什么也没做的配置"，等于啥也没学到。

Step 3: Prompt 增强 + LLM 推理

将 top-k（默认 k=2）参考 QConfig 封装入提示词，指导 LLM 基于参考推理出目标查询的配置建议：

Booster Prompt 模板 You are an experienced DBA asked to suggest possible configs for ${SYSTEM}. Your objective is to fill in the user requested parameters to optimize performance. These are the ranges of the system knobs set in the system. ${SYSTEM_KNOB_RANGES} You are provided with several reference configurations from similar prior query runs. ${REFERENCES_FORMAT} Here are the references: ${REFERENCES} **Instruction**: 1. Analyze the below **TARGET** query information. 2. Remember that you are optimizing for ${SYSTEM}. 3. Think carefully, consult references, and make suggestions. 4. Output the requested information in valid *JSON*. **TARGET**: ${SCHEMATIC} **Output in JSON**: ${FORMAT}

使用 Llama 3.1-8B-Instruct Q4-K-M 本地推理，温度 0，上下文 16384 tokens，最大输出 4096 tokens。

为什么用本地小模型？ 因为 Phase II 需要对工作负载中每个查询都调用一次 LLM（JOB 有 113 个查询）。如果用 GPT-4o，113 次调用 + 复杂 prompt = 高昂费用 + 高延迟。Llama 3.1-8B 量化版在单卡 RTX3080 上跑得很快，温度=0 保证可复现，且足以胜任"基于参考填空"这种受限任务（不需要从零创造）。

🧹 Sanitizer 三步清洗 — 为什么不可省？

LLM 输出的 JSON 不能直接喂给数据库——它可能包含幻觉字段（不存在的列名）、无效配置（参数超出合法范围）、缺失项（漏掉关键参数）。Sanitizer 是 LLM 与 DBMS 之间的"质检员"，三步逐层过滤： 

// ====== Step 6.1: 初步列表构建（三来源融合）======
initial_seeds = []
for ref in references:                  // 来源 A: 历史 QConfig 直接复用
    initial_seeds.append(ref.config)
for suggestion in llm_output:            // 来源 B: LLM 新建议
    initial_seeds.append(suggestion)
for missing_param in target_format:      // 来源 C: 缺失参数填默认
    fill_default(initial_seeds, missing_param)

// ====== Step 6.2: 探索性扩展（多样化）======
for seed in initial_seeds:
    // 静态分析：从查询谓词提取潜在索引列
    candidate_cols = extract_predicate_columns(query)
    add_index_variants(seed, candidate_cols)

    // 旋钮排列：work_mem ∈ {64MB, 128MB, 256MB}
    permute_knobs(seed, ranges)

// ====== Step 6.3: 去无效 + 精简（what-if 验证）======
for seed in extended_seeds:
    plan = postgres_explain(query, seed)   // 不真跑，用 HypoPG 估算
    if 'index_X' not in plan.used_indexes:
        seed.remove(index_X)               // 优化器没用到的索引→删除

// 泛化 → 具体化
for seed in seeds:
    seed.knobs.expand_scopes()  // 将"global enable_sort=OFF"具体化为查询级 hint

// 去重：保留计划唯一
unique_seeds = dedupe_by_plan_hash(seeds)
return unique_seeds

Step 4: Sanitizer — 配置清洗与多样化

LLM 的输出可能不完整、包含无效索引/参数或错误引用。Sanitizer 执行三步清洗：

1. 初步列表构建
三来源：LLM 建议 + QConfig 参考 + 缺失参数填充
目的：保证候选不会比"直接复用历史"更差
2. 探索性扩展
基于谓词静态分析增强索引 + 旋钮排列产生多样化计划
目的：让 LLM 没想到的配置也进入候选池
3. 去无效 + 精简
what-if 机制去除未用索引，泛化提示→具体化，保留唯一计划
目的：去掉幻觉与冗余，只保留真正有效的
💡 为什么 Sanitizer 要"两边都做"——既补充又删除？
LLM 的失败模式是双向的：(a) 创造性过头——会建议数据库根本没有的列（幻觉）；(b) 创造性不足——会过度依赖参考配置，输出和历史完全一样。Sanitizer 的"扩展"步骤强迫产生多样性（应对 b），"删除"步骤过滤无效项（应对 a）。两者结合，才能把 LLM 的"语义直觉"转化为"可执行的有效候选"。

📌 Phase II 一句话总结
对工作负载里每一个查询，从向量库里精准检索"最相似的历史最优 QConfig"作为参考，让 LLM 充当"DBA 经验推理引擎"，把历史经验"翻译"成针对当前查询的配置建议，再用 Sanitizer 兜底过滤幻觉、补充多样性。产出：每个查询的 Seeds 列表。

5 Phase III：约束组合 — 排序 + Beam Search

将每个查询的 Seeds 组合为一个全局 (holistic) 配置，解决跨查询冲突，通过 Beam Search 探索最优组合。

🎯 这一阶段要解决的问题
Phase II 给每个查询都生成了一组优秀的 Seeds。但有个根本矛盾：
  • 每个查询的最优配置不一定能同时存在。比如 Q1 想要 work_mem=1GB，Q5 想要 work_mem=64MB——一个数据库实例只能有一个 work_mem 值！
  • 索引也会冲突。Q1 想建 idx(a, b)，Q3 想建 idx(b, a)，建两个会浪费空间且优化器可能挑错。
  • 查询提示作用域不同。Q2 想 SET enable_sort=OFF，但这是会话级的，会影响所有其他查询。

Phase III 的目的就是把"每查询的局部最优 Seeds"组合成一个"全局可行且整体最优的配置"。这是一个组合优化问题——所以用 Beam Search。

⚔️ 为什么会冲突？— 一个具体场景

💡 这就是为什么需要"组合"：把 N 个查询的"局部最优 Seeds"列表，用搜索算法找到一个"全局兼容"的组合。Beam Search 是一个"束搜索"——每一步保留 top-k 候选组合（"束"），不像贪心只看一步，也不像穷举耗费太多。

🔁 Phase III 整体流程

Seed 排序策略 — 为什么不能用计划成本排序？

由于计划成本 (plan cost) 与计划质量不必然相关（尤其在查询提示干扰成本估计时），Booster 选择实际执行每个 Seed 来评估质量。

典型场景：当用户加了 SET enable_sort=OFF 提示，PostgreSQL 优化器估算的成本可能虚高（因为它认为禁用排序"很贵"），但实际执行可能因为避开了瓶颈而更快。光看 cost 会误判。

开销控制：构建替代索引 (alternate indexes) 覆盖原始索引，按计划成本递增顺序执行，配合逐查询超时策略——成本估计依然用作排序提示（先跑成本低的，更快出结果），只是不作为最终质量评判。

Beam Search 算法 (Algorithm 1)

Input:  Query-Configs QC = {..., (q_i, [c_i1,...,c_ij])}
Output: Target Configuration

1 seeds = {(q_i, best(c_i)) ∀(q_i, c_i) ∈ QC}     // 每查询选最优 seed
2 cand  = Merge(seeds)                                // 合并为全局配置
3 best  = Evaluate(cand)                              // 评估并选最优
4 while budget not exhausted do
5   q_next    = Select(best)                          // 选择待改进查询
6   alt_seeds = Rollout(q_next, best, QC)             // 生成替代 seeds
7   cand  = Merge(best, q_next, alt_seeds)            // 重新组合
8   best  = Evaluate(cand)
9 end while
10 Return best

🧩 Merge 是怎么合并的？— 真实例子

Merge 把 N 个查询的局部 Seeds 合成一个"全局配置候选"，关键是处理三类参数的合并规则：

// 输入：3 个查询的最优 Seeds（来自 Phase II）
Q1.best_seed = {work_mem: 1024, idxs: [(shipdate,returnflag)], hint: "hash_agg"}
Q2.best_seed = {work_mem: 64,   idxs: [(orderkey)],            hint: "sort_off"}
Q3.best_seed = {work_mem: 256,  idxs: [(returnflag,shipdate)], hint: "sort_on"}

// === 合并规则 ===

▶ 系统旋钮 (work_mem) — 取 min/median/max 生成多组候选
candidate_A: work_mem = 64    // min（保守）
candidate_B: work_mem = 256   // median（折中）
candidate_C: work_mem = 1024  // max（激进）

▶ 物理设计 (索引) — 取并集，去除 functional 重复
all_idxs = [
  (shipdate, returnflag),        // 来自 Q1
  (orderkey),                     // 来自 Q2
  (returnflag, shipdate)          // 来自 Q3，与 Q1 字段相同但顺序不同
]
// 由优化器决定用哪个，但保留两个供选择

▶ 查询提示 (query hints) — 全局化为查询级 hints
Q1: "/*+ HashAgg */"          // 仅作用于 Q1
Q2: "SET LOCAL enable_sort=OFF" // 仅作用于 Q2 会话
Q3: "/*+ Sort */"             // 仅作用于 Q3

// === 利用 DBMS 缓存优化 ===
// 如果 candidate_A 和 candidate_B 的 Q2 计划相同，
// 第二次运行 Q2 直接从 buffer cache 读，避免重复 IO

Merge（合并）

系统旋钮：取 Seeds 的最小/中位数/最大值生成多组配置。
物理设计：取索引并集，去除重复。
查询提示：作用域具体化为查询级。
利用 DBMS 缓存消除相同计划调用。

目的：把局部 Seeds 拼接成可执行的全局候选，处理共享参数的取值冲突

Select（选择）

优先修复合并后退化的查询（即在全局配置下比孤立测试更慢的——说明发生了冲突），然后贪心选择运行时间最长的查询来改进。通过查询无关配置追踪避免重复选择。

目的：把 Beam Search 的"探索预算"花在最有改进空间的查询上

🚀 Rollout 四种探索机制 — 各自适用什么场景？

Rollout 是给 q_next（被选中要改进的查询）生成"替代 Seeds"的过程。Booster 设计了 4 种约束探索机制（M1-M4），分别针对不同的优化机会：

Rollout — 四种约束探索机制

M1: 查询旋钮排列 (Knob Permutation)

做什么：对 Seed 的查询级旋钮进行组合枚举（如 enable_sort、enable_hashjoin）。

适用场景：当一个查询的最优计划严重依赖某些 boolean knob（如禁用某种 join）时。例如 Q1 在 enable_sort=ON/OFF 两种情况下计划完全不同。

示例：[sort=ON,hashjoin=ON] / [sort=OFF,hashjoin=ON] / [sort=ON,hashjoin=OFF] / ...

M2: 计划修复 (Plan Repair)

做什么：对比 Seed 在孤立环境和全局配置下的执行计划差异，通过 query hint 把差异节点修正回来。

适用场景：合并后产生退化的查询。比如孤立时 Q1 用 HashAgg，但全局有了 work_mem=64MB 后改成了 GroupAgg，性能掉了——M2 强制 hint 用回 HashAgg。

三步修复：(1) 启用对应优化器 flag (2) 关闭多余节点 (3) 启用缺失节点

M3: 隐藏索引 (Hidden Indexes)

做什么：故意"隐藏" Seed 当前使用的索引，让优化器找替代计划。

适用场景：多索引共存时，优化器可能挑错索引。M3 移除当前用的索引，看是否有更优替代——也可能会用别的索引扫描或全表扫，反而更快。

示例：原 Seed 用 idx(returnflag,shipdate)，M3 隐藏后强制用 idx(shipdate,returnflag)

M4: 排名 Seeds (Ranked Seeds)

做什么：从 Phase II 已经排序好的 Seed 列表里，取"估算运行时 < 当前 Seed 的"那些来尝试。

适用场景：当 q_next 的当前 Seed 在全局环境下表现不佳，但 Phase II 还提供了其他 Seeds——M4 直接试这些"备选项"。

示例：当前用 Seed_3（est=1.2s），M4 试 Seed_4(est=0.9s) / Seed_5(est=0.8s)

⏱️ 5 步 Rollout 渐进策略 — 为什么不一次性全跑？
因为 Rollout 可能产生 ~100 个候选组合，全跑要好几小时。Booster 采用渐进搜索：
步骤机制范围早停条件
1. LocalM1, M2逐查询超时局部修补成功 → 跳过后续
2. 25% SeedM3, M4Top 25% 排名 Seeds工作负载超时
3. 50% SeedM3, M4Top 50%工作负载超时
4. 75% SeedM3, M4Top 75%工作负载超时
5. 100% SeedM3, M4所有 Seeds穷尽探索

关键：如果在第 1 步就找到了好配置，后面就不再花时间——避免在次优 Seeds 上浪费预算。这是工程上的"多一次尝试就多一份成本"的现实考量。

📌 Phase III 一句话总结
用 Beam Search 在"每查询的 Seeds 列表"组成的组合空间中搜索全局最优配置。每轮迭代：(1) Evaluate 实跑当前候选，(2) Select 找最差查询作为改进目标，(3) Rollout 用 M1-M4 生成替代 Seeds，(4) Merge 重新组合。1.5 小时预算内，最终输出一个"解决了所有跨查询冲突的全局配置"，通过 Digest 注入调优器继续精炼。

6 集成机制：Parse / Link / Digest

Booster 与具体调优器解耦，通过三个 API 命令实现即插即用。

Parse（解析）

将不同格式的调优器制品统一解析为 QConfig。协调调优器支持的调优项与 Booster 推理范围的差异（如全局 vs 逐表访问方法粒度）。

Link（链接）

定义 QConfig 间的链接关系。默认基于调优轨迹的时间顺序：C1 → C2 当 C2 从 C1 探索得到。

Digest（消化）

将 Booster 的发现注入调优器继续精炼。默认暴露最优全局配置，未来可扩展暴露探索过的配置或指导调优焦点。

部署要求：操作员需提供离线环境（安全探索）+ Embedder/Prompter LLM 的 API 或本地 GPU。启动新调优会话时先调用 Booster，再运行调优器，最后让 Booster 分析新制品。

8 敏感性分析：六大消融实验

8.1 微调 vs 提示增强 (Fig.14)

对比 FT-8192、FT-16384、FT-Q 与 Booster 的提示增强方法。在三种场景下，微调均不如提示增强——上下文截断限制了学习范围，查询配置组合时冲突。

8.2 查询旋钮排列策略 (Fig.15)

Exact 场景下策略无影响（1:1 映射），漂移场景下 JoinTypes/AccessMethods/Sort 策略均优于 None。All 策略最优，充分利用所有多样化计划机会。

8.3 搜索时间 (Fig.16)

更多搜索时间 → 更好配置，但收益递减。Exact 场景 1.5hr 后稳定，漂移场景持续微改善。默认 1.5hr 是合理的折中。

8.4 Embedder-Prompter 模型 (Fig.17)

在跨 Schema 场景下模型选择影响最大（差异约 17s）。更大 Prompter (o3-mini) 比 Qwen3-1.7B 好 32%，但成本更高。简单场景无需大型模型。

8.5 Rollout 策略 (Fig.18)

M2(计划修复)和 M3(隐藏索引)单独效果有限；M1(旋钮排列)和 M4(排名 Seeds)更有效。All 组合最优。Template Drift 下查询局部变化 (M1,M2) 优于物理变化 (M3,M4)。

8.6 输入数据量 (Fig.19)

4-Input 通常优于 1-Input（更广查询洞察范围），但 λ-Tune+AS 例外（1-Input 提供更多样化查询模板参考）。更多数据不总是更好，取决于制品多样性。

9 核心洞察与局限性

洞察 1: 查询粒度是关键

工作负载级适应无法组合不同配置的最佳方面。Booster 证明了按查询推理 + 组合的范式能释放历史知识的全部潜力，实现 "Total Knowledge Remix"。

洞察 2: LLM 是推理引擎而非知识库

Booster 不依赖 LLM 的预训练知识来生成配置，而是将其作为推理引擎——在精心策划的历史参考上进行推理。这解释了为何提示增强远优于微调。

洞察 3: 链接追踪是检索的灵魂

最相似的 QConfig 往往不含调优指导，沿链接追踪到最优下游 QConfig 是关键创新。这一设计将 RAG 的"检索-增强"范式从静态片段提升为动态经验链。

洞察 4: 约束探索弥补 LLM 局限

LLM 可能生成局部次优配置。Booster 的 Sanitizer 和 Rollout 机制 (M1–M4) 通过系统性探索弥补这一不足，在 LLM 建议邻域中发现更优方案。

局限性 1: 历史过拟合

当 Proto-X 过拟合 SF1 配置（如关闭哈希聚合、忽略覆盖索引）时，Booster 被困于局部最优。论文坦承需要调优器同时考虑稳定性和泛化性目标。

局限性 2: 仅验证 OLAP

所有实验仅覆盖 OLAP 工作负载，OLTP 场景下的效果未验证。

局限性 3: 仅验证 PostgreSQL

所有实验在单一 DBMS 上进行，跨数据库系统的泛化能力未知。

局限性 4: 何时调用 Booster

依赖操作员判断何时触发 Booster，缺乏自动化的成本-收益平衡机制。

10 总结与未来展望

核心贡献

  1. 提出 Booster 框架，通过查询级洞察重混历史知识，辅助现有调优器适应环境变化
  2. 设计 QConfig + Schematic + IV 体系，将非结构化调优经验标准化为可搜索的向量表示
  3. 创新性地使用 链接追踪机制，从最相似 QConfig 获取最优下游配置作为 LLM 参考
  4. 设计 Sanitizer + Beam Search + 四种 Rollout 机制，系统性地组合查询级 Seeds 为全局配置
  5. 在 7 大场景 × 4 种调优器 上全面验证，最高 74% 性能提升和 4.7× 速度提升

未来方向

  • 构建考虑稳定性和泛化性目标的调优器（避免过拟合）
  • 开发自动化决定何时调用 Booster的机制
  • 基于查询动态选择最优 QConfig 数量来增强提示
  • 基于泛化性和成本需求选择最优 LLM
  • 扩展至OLTP 工作负载和多 DBMS场景
  • 支持 Digest 暴露更多制品（如探索过的配置、调优焦点指导）
一句话总结：Booster 证明了 LLM 不必替代调优器——当作为"经验推理引擎"与查询级检索增强和约束组合结合时，LLM 能解锁历史知识中隐藏的巨大潜力，让现有调优器在环境变化时更快找到更好的配置。

深度问答

Q1: 为什么 Booster 选择提示增强而非微调？
两个原因：(1) 微调需要截断上下文到固定长度（8192/16384 tokens），限制了 LLM 能学习的查询范围；(2) 即使采用查询级微调 (FT-Q)，不同查询的配置在组合为全局配置时会发生冲突。而提示增强允许 LLM 在推理时参考精心策划的历史经验，通过 Sanitizer 和 Beam Search 系统性地解决组合冲突。实验表明提示增强在所有场景下均优于微调。
Q2: 链接追踪 (Link Following) 为什么重要？
向量检索返回的最相似 QConfig 往往来自同一查询在初始（如 stock）配置下的记录——这个 QConfig 不包含任何调优指导（没有有用的旋钮设置、没有推荐的索引）。沿链接追踪到最优下游 QConfig，才能获取调优器经过多轮探索后发现的真正有效的配置。这一设计将检索范式从"找最相似的状态"升级为"找最相似状态下最好的调优结果"。
Q3: Booster 如何处理跨 Schema 迁移中的查询匹配？
关键在于匿名化 Schematic。Booster 不仅使用原始 SQL/Plan，还生成匿名化版本（表名、列名替换为占位符），这样不同 Schema 中结构相似的查询（如两个多表 JOIN 聚合查询）即使表名不同，其匿名化后的表示也会在向量空间中相近，从而实现跨 Schema 的语义匹配。
Q4: Booster 为什么选择小型本地 LLM 而非 GPT-4？
主要出于成本和部署考量：(1) 查询级提示需要为每个查询调用 LLM（49–113 次），使用 GPT-4 成本高昂；(2) Booster 依赖离线环境安全探索，本地 LLM 无需网络传输敏感数据；(3) 实验表明 Llama 3.1-8B 已能配合检索增强产生有效建议，Sanitizer 和 Beam Search 弥补了 LLM 输出的不足。敏感性实验也显示在简单场景下小型模型已足够，仅在跨 Schema 场景才需要更大型模型。
Q5: Beam Search 的四种 Rollout 机制各自解决什么问题？
M1 (查询旋钮排列)：解决 LLM 可能遗漏有效计划变体的问题，通过系统性排列产生多样化候选。
M2 (计划修复)：解决合并为全局配置后查询计划偏离独立 Seed 的问题，通过修改查询提示恢复原计划。
M3 (隐藏索引)：解决索引集组合损害成本优化器判断的问题，通过移除已用索引探索替代路径。
M4 (排名 Seeds)：解决仅使用最优 Seed 可能错过更好组合的问题，引入其他低运行时间 Seeds。
