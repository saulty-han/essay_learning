1 研究动机：调优器的适应性困境

数据库管理系统 (DBMS) 暴露了数万亿种可能的配置选项（旋钮、索引、查询提示），同时应用负载不断演变。 现有自动调优器虽然能针对固定工作负载找到优良配置，但在环境变化（负载漂移、跨 Schema 迁移）时却力不从心。

刚性与脆弱性 (Rigidity & Brittleness)

基于成本的调优器：缺乏将先验洞察注入搜索过程的机制——即便历史经验揭示了两个索引的不良交互，也无法优先排除它们。

基于 ML 的调优器：环境变化导致内部模型表示损坏（如 Q1 不再存在），必须重新优化，而预训练因目标环境差异和重新评估开销过大而不可行。

工作负载粒度问题 (Workload Granularity)

现有调优器在工作负载级别做适应性调整，使用负载级遥测来映射历史配置。但这种方式无法跨配置组合单个查询的洞察——例如 C1 优化了 Q1，C2 优化了 Q2，调优器无法将两者最佳方面组合起来。

LLM 调优器同样忽视了按查询逐个增强提示的机会。

关键实验证据 (Fig.1)：在 Transfer (TPC-H → DSB) 和 Drift (DSB 50% 参数变化) 场景下，"Total Knowledge Remix"（完全重组历史知识）比"Continue From History"（从历史继续调优）发现性能提升 13%–74%。这揭示了历史知识中蕴藏的巨大潜力。

2 框架概览：Booster 三阶段架构

Booster 作为现有调优器的"加速器"插件：分析历史制品仓库，将发现注入调优器，被注入的调优器进一步精炼配置并存回制品。

图：Booster 三阶段架构流程图 — Phase I (紫) → Phase II (橙) → Phase III (绿)

核心思想：Booster 不替代现有调优器，而是加速其适应环境变化。它通过查询级（而非工作负载级）的洞察重混历史知识，实现了"Total Knowledge Remix"。

3 Phase I：经验分析 — QConfig 构建与向量化

Booster 监控调优器的制品仓库，将非结构化的历史经验标准化为可搜索的 QConfig 对象，并向量化存入向量数据库。

QConfig 数据结构

每个 QConfig 封装了一个查询在某配置下的行为，包含：

Schema 元数据
表名、列类型、统计信息、已有索引
查询信息
SQL 文本 / 查询模板 / 执行计划
配置信息
系统旋钮、索引、查询提示
链接指针
指向同一查询的下一个 QConfig (Next)

Schematic 构建 — 多视角向量化

每个 QConfig 生成多种 Schematic（文本表示），涵盖从高层到低层的查询语义：

Schematic 类型Schema 部分查询部分变体
SQL 原文Schema 元数据SQL 文本原始 / 匿名化
查询模板Schema 元数据参数化 SQL原始 / 匿名化
执行计划Schema 元数据Plan 文本原始 / 匿名化

每种 Schematic 通过 Voyage 3 Large 嵌入模型生成一个 Identity Vector (IV)，共 6 个 IV（3 类型 × 2 变体）。

为什么需要链接？ 最语义相似的 QConfig 往往来自同一查询在初始配置下的记录——但它不包含任何有用的调优指导。通过链接追踪到最优下游 QConfig，才能获取真正有价值的配置参考。

4 Phase II：引导推荐 — 检索 + LLM 生成 + Sanitizer

当环境变化时，Booster 对工作负载中每个查询，从向量数据库检索相关历史经验，增强 LLM 提示词生成配置建议，再经 Sanitizer 清洗得到可行的查询 Seeds。

Step 1: 查询向量化 + 相似检索

对目标查询生成 Schematic 和 IVs（与 Phase I 相同流程），然后在向量数据库中按欧氏距离检索同类 Schematic 的最相似 QConfig。关键设计：仅在同类型 Schematic 间比较，避免跨类型误排。

Step 2: 链接追踪 — 获取最优下游 QConfig

从每个检索到的 QConfig 出发，沿链接追踪到最优下游 QConfig（运行时最短的那个）。这确保 LLM 参考的是历史中表现最好的配置，而非初始状态的"空"配置。

Step 3: Prompt 增强 + LLM 推理

将 top-k（默认 k=2）参考 QConfig 封装入提示词，指导 LLM 基于参考推理出目标查询的配置建议：

Booster Prompt 模板 You are an experienced DBA asked to suggest possible configs for ${SYSTEM}. Your objective is to fill in the user requested parameters to optimize performance. These are the ranges of the system knobs set in the system. ${SYSTEM_KNOB_RANGES} You are provided with several reference configurations from similar prior query runs. ${REFERENCES_FORMAT} Here are the references: ${REFERENCES} **Instruction**: 1. Analyze the below **TARGET** query information. 2. Remember that you are optimizing for ${SYSTEM}. 3. Think carefully, consult references, and make suggestions. 4. Output the requested information in valid *JSON*. **TARGET**: ${SCHEMATIC} **Output in JSON**: ${FORMAT}

使用 Llama 3.1-8B-Instruct Q4-K-M 本地推理，温度 0，上下文 16384 tokens，最大输出 4096 tokens。

Step 4: Sanitizer — 配置清洗与多样化

LLM 的输出可能不完整、包含无效索引/参数或错误引用。Sanitizer 执行三步清洗：

1. 初步列表构建
三来源：LLM 建议 + QConfig 参考 + 缺失参数填充
2. 探索性扩展
基于谓词静态分析增强索引 + 旋钮排列产生多样化计划
3. 去无效 + 精简
what-if 机制去除未用索引，泛化提示→具体化，保留唯一计划

5 Phase III：约束组合 — 排序 + Beam Search

将每个查询的 Seeds 组合为一个全局 (holistic) 配置，解决跨查询冲突，通过 Beam Search 探索最优组合。

Seed 排序策略

由于计划成本 (plan cost) 与计划质量不必然相关（尤其在查询提示干扰成本估计时），Booster 选择实际执行每个 Seed 来评估质量。

开销控制：构建替代索引 (alternate indexes) 覆盖原始索引，按计划成本递增顺序执行，配合逐查询超时策略。

Beam Search 算法 (Algorithm 1)

Input:  Query-Configs QC = {..., (q_i, [c_i1,...,c_ij])}
Output: Target Configuration

1 seeds = {(q_i, best(c_i)) ∀(q_i, c_i) ∈ QC}     // 每查询选最优 seed
2 cand  = Merge(seeds)                                // 合并为全局配置
3 best  = Evaluate(cand)                              // 评估并选最优
4 while budget not exhausted do
5   q_next    = Select(best)                          // 选择待改进查询
6   alt_seeds = Rollout(q_next, best, QC)             // 生成替代 seeds
7   cand  = Merge(best, q_next, alt_seeds)            // 重新组合
8   best  = Evaluate(cand)
9 end while
10 Return best

Merge（合并）

系统旋钮：取 Seeds 的最小/中位数/最大值生成多组配置。
物理设计：取索引并集，去除重复。
利用 DBMS 缓存消除相同计划调用。

Select（选择）

优先修复合并后退化的查询（冲突），然后贪心选择运行时间最长的查询来改进。通过查询无关配置追踪避免重复选择。

Rollout — 四种约束探索机制

M1: 查询旋钮排列

排列 Seed 的查询旋钮（如关闭排序、强制 NL Join），产生多样化计划。

M2: 计划修复

对比 Seed 独立计划 vs 全局配置下的计划，通过修改查询提示恢复/消除差异节点。

M3: 隐藏索引

移除 Seed 已使用的索引，探索替代高效计划（索引集组合可能损害成本优化器）。

M4: 排名 Seeds

选择估计运行时间小于当前 Seed 的其他排名 Seeds。

5 步 Rollout 策略：Local (M1,M2) → 25% Seed (M3,M4) → 50% Seed → 75% Seed → 100% Seed。逐步扩大探索范围，若早期发现优良配置则避免后续次优 Seeds。

6 集成机制：Parse / Link / Digest

Booster 与具体调优器解耦，通过三个 API 命令实现即插即用。

Parse（解析）

将不同格式的调优器制品统一解析为 QConfig。协调调优器支持的调优项与 Booster 推理范围的差异（如全局 vs 逐表访问方法粒度）。

Link（链接）

定义 QConfig 间的链接关系。默认基于调优轨迹的时间顺序：C1 → C2 当 C2 从 C1 探索得到。

Digest（消化）

将 Booster 的发现注入调优器继续精炼。默认暴露最优全局配置，未来可扩展暴露探索过的配置或指导调优焦点。

部署要求：操作员需提供离线环境（安全探索）+ Embedder/Prompter LLM 的 API 或本地 GPU。启动新调优会话时先调用 Booster，再运行调优器，最后让 Booster 分析新制品。

8 敏感性分析：六大消融实验

8.1 微调 vs 提示增强 (Fig.14)

对比 FT-8192、FT-16384、FT-Q 与 Booster 的提示增强方法。在三种场景下，微调均不如提示增强——上下文截断限制了学习范围，查询配置组合时冲突。

8.2 查询旋钮排列策略 (Fig.15)

Exact 场景下策略无影响（1:1 映射），漂移场景下 JoinTypes/AccessMethods/Sort 策略均优于 None。All 策略最优，充分利用所有多样化计划机会。

8.3 搜索时间 (Fig.16)

更多搜索时间 → 更好配置，但收益递减。Exact 场景 1.5hr 后稳定，漂移场景持续微改善。默认 1.5hr 是合理的折中。

8.4 Embedder-Prompter 模型 (Fig.17)

在跨 Schema 场景下模型选择影响最大（差异约 17s）。更大 Prompter (o3-mini) 比 Qwen3-1.7B 好 32%，但成本更高。简单场景无需大型模型。

8.5 Rollout 策略 (Fig.18)

M2(计划修复)和 M3(隐藏索引)单独效果有限；M1(旋钮排列)和 M4(排名 Seeds)更有效。All 组合最优。Template Drift 下查询局部变化 (M1,M2) 优于物理变化 (M3,M4)。

8.6 输入数据量 (Fig.19)

4-Input 通常优于 1-Input（更广查询洞察范围），但 λ-Tune+AS 例外（1-Input 提供更多样化查询模板参考）。更多数据不总是更好，取决于制品多样性。

9 核心洞察与局限性

洞察 1: 查询粒度是关键

工作负载级适应无法组合不同配置的最佳方面。Booster 证明了按查询推理 + 组合的范式能释放历史知识的全部潜力，实现 "Total Knowledge Remix"。

洞察 2: LLM 是推理引擎而非知识库

Booster 不依赖 LLM 的预训练知识来生成配置，而是将其作为推理引擎——在精心策划的历史参考上进行推理。这解释了为何提示增强远优于微调。

洞察 3: 链接追踪是检索的灵魂

最相似的 QConfig 往往不含调优指导，沿链接追踪到最优下游 QConfig 是关键创新。这一设计将 RAG 的"检索-增强"范式从静态片段提升为动态经验链。

洞察 4: 约束探索弥补 LLM 局限

LLM 可能生成局部次优配置。Booster 的 Sanitizer 和 Rollout 机制 (M1–M4) 通过系统性探索弥补这一不足，在 LLM 建议邻域中发现更优方案。

局限性 1: 历史过拟合

当 Proto-X 过拟合 SF1 配置（如关闭哈希聚合、忽略覆盖索引）时，Booster 被困于局部最优。论文坦承需要调优器同时考虑稳定性和泛化性目标。

局限性 2: 仅验证 OLAP

所有实验仅覆盖 OLAP 工作负载，OLTP 场景下的效果未验证。

局限性 3: 仅验证 PostgreSQL

所有实验在单一 DBMS 上进行，跨数据库系统的泛化能力未知。

局限性 4: 何时调用 Booster

依赖操作员判断何时触发 Booster，缺乏自动化的成本-收益平衡机制。

10 总结与未来展望

核心贡献

  1. 提出 Booster 框架，通过查询级洞察重混历史知识，辅助现有调优器适应环境变化
  2. 设计 QConfig + Schematic + IV 体系，将非结构化调优经验标准化为可搜索的向量表示
  3. 创新性地使用 链接追踪机制，从最相似 QConfig 获取最优下游配置作为 LLM 参考
  4. 设计 Sanitizer + Beam Search + 四种 Rollout 机制，系统性地组合查询级 Seeds 为全局配置
  5. 在 7 大场景 × 4 种调优器 上全面验证，最高 74% 性能提升和 4.7× 速度提升

未来方向

  • 构建考虑稳定性和泛化性目标的调优器（避免过拟合）
  • 开发自动化决定何时调用 Booster的机制
  • 基于查询动态选择最优 QConfig 数量来增强提示
  • 基于泛化性和成本需求选择最优 LLM
  • 扩展至OLTP 工作负载和多 DBMS场景
  • 支持 Digest 暴露更多制品（如探索过的配置、调优焦点指导）
一句话总结：Booster 证明了 LLM 不必替代调优器——当作为"经验推理引擎"与查询级检索增强和约束组合结合时，LLM 能解锁历史知识中隐藏的巨大潜力，让现有调优器在环境变化时更快找到更好的配置。

深度问答

Q1: 为什么 Booster 选择提示增强而非微调？
两个原因：(1) 微调需要截断上下文到固定长度（8192/16384 tokens），限制了 LLM 能学习的查询范围；(2) 即使采用查询级微调 (FT-Q)，不同查询的配置在组合为全局配置时会发生冲突。而提示增强允许 LLM 在推理时参考精心策划的历史经验，通过 Sanitizer 和 Beam Search 系统性地解决组合冲突。实验表明提示增强在所有场景下均优于微调。
Q2: 链接追踪 (Link Following) 为什么重要？
向量检索返回的最相似 QConfig 往往来自同一查询在初始（如 stock）配置下的记录——这个 QConfig 不包含任何调优指导（没有有用的旋钮设置、没有推荐的索引）。沿链接追踪到最优下游 QConfig，才能获取调优器经过多轮探索后发现的真正有效的配置。这一设计将检索范式从"找最相似的状态"升级为"找最相似状态下最好的调优结果"。
Q3: Booster 如何处理跨 Schema 迁移中的查询匹配？
关键在于匿名化 Schematic。Booster 不仅使用原始 SQL/Plan，还生成匿名化版本（表名、列名替换为占位符），这样不同 Schema 中结构相似的查询（如两个多表 JOIN 聚合查询）即使表名不同，其匿名化后的表示也会在向量空间中相近，从而实现跨 Schema 的语义匹配。
Q4: Booster 为什么选择小型本地 LLM 而非 GPT-4？
主要出于成本和部署考量：(1) 查询级提示需要为每个查询调用 LLM（49–113 次），使用 GPT-4 成本高昂；(2) Booster 依赖离线环境安全探索，本地 LLM 无需网络传输敏感数据；(3) 实验表明 Llama 3.1-8B 已能配合检索增强产生有效建议，Sanitizer 和 Beam Search 弥补了 LLM 输出的不足。敏感性实验也显示在简单场景下小型模型已足够，仅在跨 Schema 场景才需要更大型模型。
Q5: Beam Search 的四种 Rollout 机制各自解决什么问题？
M1 (查询旋钮排列)：解决 LLM 可能遗漏有效计划变体的问题，通过系统性排列产生多样化候选。
M2 (计划修复)：解决合并为全局配置后查询计划偏离独立 Seed 的问题，通过修改查询提示恢复原计划。
M3 (隐藏索引)：解决索引集组合损害成本优化器判断的问题，通过移除已用索引探索替代路径。
M4 (排名 Seeds)：解决仅使用最优 Seed 可能错过更好组合的问题，引入其他低运行时间 Seeds。
四者互补，All 组合下效果最佳。
