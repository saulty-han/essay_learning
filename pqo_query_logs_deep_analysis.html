1研究动机：PQO 的两种任务与解耦思想

参数化查询（如存储过程）被反复执行、仅参数绑定不同。对复杂查询，每次都重新优化（Opt-Always）开销巨大；而只优化第一个实例并缓存其计划（Opt-Once）虽省优化时间，但缓存计划对后续实例可能「任意次优」。PQO 是折中：用较少的缓存计划，把执行次优性压到尽量低。它包含两个任务：

populateCache（填缓存）

为某个参数化查询找出「要缓存的一组计划」。运行在离线阶段，结果安装进计划缓存。

getPlan（取计划）

新查询实例到来时，从缓存计划里高效挑出最适合它的那一个。它在查询执行的关键路径上，因此延迟是关键指标。

核心主张 ①：把 populateCache 与 getPlan 架构解耦。 此前多数在线 PQO 把两者交织在一起：getPlan 失败时回退优化器、并动态增删缓存计划。这带来两大问题——① 运营上不可预测、难调试（计划集一直在变）；② 常需昂贵的优化器/重算（recost）调用，导致 getPlan 的高均值与尾延迟。解耦后，getPlan 只从缓存里选、不回退优化器，立刻解决延迟；populateCache 作为离线步骤，更可控、可调试，还允许 DBA 手动加计划。

核心主张 ② 与 ③

② 用查询日志驱动：现代 DBMS 本就高效记录查询实例（参数、执行时间、CPU 时间）。populateCache 用日志在「巨大的选择性空间」里聚焦到 workload 相关区域；getPlan 用日志训练监督 ML 模型。
③ 用执行代价驱动 getPlan：优化器代价估计常不准（基数估计误差、代价模型简化），作者把 getPlan 重新表述为「情境多臂老虎机（CMAB）」问题，用每次执行的真实代价反馈来学习选最低执行代价的计划。

对先前在线技术的批判：它们假设计划代价函数线性/单调，而复杂 SQL 下这些假设常失败 → getPlan 次优性很高；或假设更保守 → 频繁回退优化器 → getPlan 延迟高。本文的 ML 方法对任意复杂 SQL 都适用、且不做代价行为假设。

3问题定义：次优性指标与两个目标

论文用「次优性（sub-optimality）」而非绝对代价来度量，便于跨查询聚合（几何均值或 95 分位）。

3.1 populateCache 的次优性（公式 1）

给定 workload 𝒲 与计划预算 K，选出计划集 𝒫 = {p₁…p_K} 使聚合次优性最小：

S_populateCache(𝒫, @x) = min_{p∈𝒫} C_opt(p,@x) / C_opt(p_opt(@x),@x)

分子假设一个「完美 getPlan」（从 𝒫 中选最低代价计划），分母是理想策略 Opt-Always 的代价。此外还度量找出计划的总时间、optimize/recost 调用次数。

3.2 getPlan 的两个目标

Objective 1（优化器代价）：从 𝒫 中选使优化器估计代价最小的计划（公式 2）。

Objective 2（执行代价）：从 𝒫 中选使真实执行代价最小的计划（公式 3）。

Obj1: p₀ = arg min_{p∈𝒫} C_opt(p,@x)    |    Obj2: p₀ = arg min_{p∈𝒫} ExecCost(p,@x)

度量：次优性

几何均值或 95 分位，越小越好（1.0 为最优）。

度量：getPlan 延迟

选计划所花时间，报告均值与尾分位。

度量：标注/训练开销

生成标签与训练模型的时间。

关键区分

整体次优性 = populateCache 次优性 × getPlan 次优性。由于 getPlan 在关键路径上，效率（延迟）与次优性同等重要——这正是先前在线技术顾此失彼的地方。

4基于优化器代价的 getPlan：分类 vs 回归

给定缓存的 K 个计划，getPlan 要选代价最低的那个。最准确的方法是「对每个计划 recost 一下取最小」，但 recost 对复杂查询常需 10ms~数百 ms，太贵（且需支持任意复杂 SQL 的 recost 特性）。作者改用监督 ML：用极小的内存 footprint 与极快的推理来近似。

4.1 特征：选择性向量

用「每个参数化谓词的选择性」组成的向量作输入特征。直觉：模型把高维选择性空间划分区域，每个区域映射到一个缓存计划。这比先前工作好得多，因为 getPlan 完全不需要调用优化器（选择性可用直方图等统计信息高效获得，约 25µs/选择性）。也尝试了直接用参数绑定（解析后的范围/类别谓词），准确性略差于选择性特征。

4.2 分类 formulation（多类）

把 getPlan 看作 K 类分类：K 个计划 = K 个类，训练数据由 workload 中每个实例标注「最佳计划 id」构成。模型学习在特征空间（选择性）上划分区域。

局限（论文 Figure 3）：分类的目标是「最大化分类准确率」，与我们的次优性指标并不对齐。因为把实例 @x 错分为计划 p_a 而非真正最优的 p_b，其代价损失取决于 p_a、p_b、@x——这是「样例依赖的代价敏感分类」，多类情形很难处理。Figure 3 的例子中，把标为 p4 的点分成 p3 可能比分成 p2 的次优性更高，但普通分类器无法区分。

4.3 回归 formulation（每计划一个回归器）

为每个缓存计划各训练一个回归模型，预测该计划在给定特征下的优化器估计代价；对新实例调用 K 个回归器取预测代价最小的计划（Figure 2）。这是「对每个计划 recost」的低开销近似（快 1~2 个数量级），且拥有所有计划的完整代价信息，不像分类只用「最佳计划」标签。

4.4 训练与「分类 vs 回归」

两种形式的标注数据其实相同：每个缓存计划在 workload 实例上的代价（回归直接用，分类据此生成最佳计划标签）。训练时间本身很小（约 100ms），主要开销在标注（对每个计划 recost）。实验用决策树、XGBoost 等树模型。结论：小内存/小训练量下，回归次优性更好（尤其 95 分位）；getPlan 时间仅略高于分类。

5基于执行代价的 getPlan：情境多臂老虎机

无论分类还是回归，上面的 getPlan 都在最小化「优化器估计代价」，但估计代价 ≠ 执行代价（基数估计误差、代价模型无法刻画执行环境）。作者把 getPlan 重新表述为情境多臂老虎机（CMAB）：臂 = 缓存计划，上下文 = 选择性（或参数绑定）向量，奖励 = 该实例用所选计划的（负）执行代价。随着实例被处理，老虎机不断学习选更低执行代价的计划。

图 5（Algorithm 1 Bootstrapped Thompson Sampling 环路）：每计划维护 C 个回归模型；对新实例各随机抽 1 个模型预测代价，选最小者执行；用真实执行代价以 Gamma 加权更新这些模型。

5.1 Algorithm 1：在线 Bootstrapped Thompson Sampling

对每个计划维护 C 个监督回归模型。每来一个实例 @x：① 对每个计划 i，从 C 个模型中随机选 1 个，预测其执行代价 ˆA_i；② 选 a = argmin_i ˆA_i 执行，得到真实代价 A；③ 把观测 {(φ, A)} 加入「计划 a 对应模型」的历史，并对每个模型 j 以 Gamma(1,1) 采样的权重更新。随机选模型 → 探索；训练充分后选最小预测代价 → 利用。

5.2 避免初始次优 & 适应变化

初始兜底：老虎机在初始阶段随机选计划，往往比「选优化器估计代价最低的计划」更差。缓解策略：前 50 个实例先用优化器代价基线（或并行执行所有 K 计划取最先完成的）决策，之后切换到老虎机——既避免初期风险，又保留老虎机收益。

适应计划集变化：新增计划时老虎机自动适应（暂时次优但快速收敛）；删除计划只需忽略该「臂」。这比监督 ML 需重训更灵活。

6populateCache 算法：贪心搜索与三阶段

populateCache 的目标是用很少的计划覆盖整个 workload、使聚合次优性低。但 K-set 识别对最大代价次优性指标已被证明是 NP 完全的；朴素实现（2000 实例、100 计划、K=5）可能超过 10 小时。因此论文用启发式把每阶段开销降下来。

阶段 1 计划收集

对 workload 子集调用优化器，最坏开销 n·T_opt；收集到 m 个不同计划。

阶段 2 计划重算

对每个候选计划在 n 个实例上 recost，得「plan-recost 矩阵」，最坏开销 n·m·T_recost。

阶段 3 K-set 识别

用矩阵确定预算 K 下的缓存计划集；朴素最坏开销需评估约 7500 万候选集，不现实。

6.1 K-set 识别：贪心搜索

从空集开始，每轮贪心加入「能使聚合次优性指标最小化的一个计划」，直到达到预算 K。复杂度 O(K·m·T_metric)，在 plan-recost 矩阵算好后极快：200 计划、2000 实例、K=10、几何均值指标下，贪心运行 < 1 秒。

6.2 计划收集 / 重算：只用子集

为降低这两阶段开销而不显著损质：计划收集从日志随机选约 100–200 个实例优化即可（收集到的计划集质量很快稳定）；计划重算随机选约 200–400 个实例 recost 即可获得足够信息来决策 K 个计划。

应对数据 / workload 变化：监督 ML 假设数据 workload 不变；显著变化时需重跑 populateCache 并重训 getPlan 模型（先生成标签再训练）。老虎机则能持续重训自适应（见 6.5）。数据变化可由 DBMS 的统计更新触发机制探测，workload 漂移可用现有漂移检测技术追踪。

7实验评估：TPC-DS / TPC-H / REAL1

实验在 Microsoft SQL Server 2017 上，用 TPC-DS(100GB)、TPC-H skewed(10GB)、真实库 REAL1(97GB)，共 25 个参数化查询（多数含 UNION/INTERSECT/EXISTS/外连接等复杂结构，5–10 个 join）。每模板生成 2000 个实例，参数谓词 3–16 个（中位 9）。模型默认 XGBoost、16KB 内存、选择性特征。

对比维度结论
getPlan 时间 vs 次优性（图 5）分类/回归监督 ML 在两个维度上都显著优于所有先前技术，有时好几个数量级；Opt-Once 次优性最差但最快；MR 均值快但次优性高；SCR 近最优但 getPlan 时间极高。
分类 vs 回归（图 6/6.2.2）小训练量（100 例）下，分类尾次优性可达 100，回归仅约 5%；回归在高分位更稳。
populateCache（图 8）K=6 时几何均值次优性 < 1.2；100-200 实例优化、200-400 实例 recost 即足够好。
端到端（图 9）回归模型训练数据 −=100, .=200 即达合理精度；初始优化器开销很快被摊薄（训练后不再有优化器调用）。

7.1 优化器代价 getPlan（6.2）

对每模板用 populateCache 找 4–7 个好计划，recost 全部 2000 实例得到 plan-recost 矩阵；1600 实例训练/构造逻辑，400 实例测试，重复 10 次。结果与 Opt-Once、MR、PCM、SCR、Density 对比：监督 ML 在次优性剖面最好，且 getPlan 时间最小、方差最低（所有优化器交互都在离线训练阶段）。

7.2 执行代价 getPlan（6.5）

对每模板给 2000 实例序列，策略每步选一个计划执行、以执行代价作反馈。对比四种：optimizer-cost-baseline、bandit-random、bandit-mitigated（前 50 用优化器代价，之后切老虎机）、bandit-exec-bootstrap（前 50 并行执行所有计划取最优）。bandit 类最终都收敛到比基线更低的执行次优性。

7.3 老虎机的鲁棒性（图 11）

workload 漂移：构造「先小选择性 200、中 300、大选择性」的对抗序列，老虎机在分布切换时尾次优性短暂上升但很快重新适应。
计划集漂移：500 实例后随机删一半计划、1000 后加回，bandit-mitigated 利用先前反馈迅速回到原性能。
Thompson Sampling 重要：TS/UCB 优于 ε-greedy；选 TS 因其 getPlan 效率优于 UCB。
噪声鲁棒：即使给奖励乘 Gamma 噪声，老虎机仍随时间改善，只是收敛变慢。

8敏感性分析：模型大小、训练量、特征

论文用回归模型做敏感性实验（分类趋势类似），逐一变动模型大小、训练量、模型类型、特征。

8.1 模型内存（图 7A）

2KB→100KB 次优性持续改善但收益递减；默认 16KB 可接受（一个缓存计划本身至少 100KB）。

8.2 训练量（图 7B）

100→1600 例次优性显著改善；400 例即可在训练成本与次优性间取得合理平衡，更多只在极高分位有微小收益。

8.3 模型类型（图 7C）

树模型（回归树及其 boosting 版 XGBoost）显著优于线性、SVM 等简单模型。

8.4 特征（图 7D）

选择性特征紧凑、准确、易得；仅用参数绑定也能得到合理模型，但未发现作为额外特征有显著优势。

10总结、贡献与局限

论文重新审视 PQO，提出三个关键思想：① 解耦 populateCache 与 getPlan；② 用历史查询日志驱动两者；③ 把 getPlan 拓展到「执行代价」导向（情境老虎机）。

主要贡献：· 解耦架构带来灵活性与可调试性，对生产至关重要；· 把 populateCache 表述为基于日志的离线搜索，贪心算法用很少计划（≤6）取得低几何均值次优性（≈1.2）且可扩展到大量实例；· getPlan 用极小 ML 模型同时取得低延迟与低次优性，且对任意 SQL 不做代价行为假设；· 首次把 getPlan 表述为基于执行代价的 CMAB 问题；· 在合成与真实 workload 上做了与多种先前 PQO 技术及商业 DBMS 方案的全面对比。

局限与未来工作

· 监督 ML 假设数据/workload 不变，显著变化需重跑离线步骤；· populateCache 依赖候选计划质量，极端高维选择性空间仍具挑战；· 执行代价老虎机需在线反馈回路，落地需配套执行采集；· workload 漂移检测与自动重训留作未来工作。

Q1：为什么「解耦」这么重要？
因为 getPlan 在查询关键路径上，若它失败时回退优化器（如 SCR），getPlan 延迟会很高且方差大。解耦后 getPlan 只从缓存里选、不回退，延迟恒定且小；populateCache 离线跑，可调试、可让 DBA 干预。
Q2：分类和回归两种 getPlan，到底用哪个？
默认推荐回归：为每个缓存计划训练一个回归器预测其代价、取最小。它在小训练量下尾次优性显著优于分类（100 例时分类尾次优性可达 100，回归约 5%），且对「样例依赖代价」更鲁棒。
Q3：执行代价老虎机会不会一开始就把查询搞得很慢？
会——随机探索阶段比优化器代价基线差。论文用「前 50 个实例先用优化器代价基线（或并行执行所有计划取最优）决策，再切老虎机」来缓解，既躲开初期风险又保留老虎机长期收益。
Q4：和后来的 Kepler 什么关系？
Kepler 沿用了本文的「解耦 + 查询日志」架构，但把候选生成升级为 RCE（扰动基数）、把标签从「优化器代价」换成「真实执行延迟」、把模型换成带 SNGP 不确定性的神经网络并加兜底。本文是 Kepler 的直接思想源头。

摘要

参数化查询优化（PQO）须解决两个问题：为参数化查询识别相对较少的待缓存计划（populateCache），以及高效地为该查询的任意实例选出要执行的最佳缓存计划（getPlan）。我们的方法把这两个决策解耦。我们把 populateCache 形式化为一个优化问题，目标是识别出一组计划，使日志中查询的优化器估计代价最小，并给出高效算法。对于 getPlan，我们利用查询日志训练机器学习（ML）模型，从缓存计划中挑选优化器估计代价最低的计划。我们用基准与真实 workload 中的复杂参数化查询做了大量实验。我们的 populateCache 算法即使对复杂查询、用较少计划也取得低几何均值次优性（≈1.2），且能扩展到大型查询日志。我们基于 ML 的 getPlan 技术的平均延迟（≈210µs）比先前 PQO 技术快一到四个数量级；平均次优性低（≈1.05），95 分位次优性比先前技术低 1% 到 25%。最后，我们给出一个利用查询日志中执行时间信息、以规避优化器代价估计不准确的 getPlan 高效算法。

1 引言（节选）

数据库应用广泛使用参数化查询（同一 SQL 反复执行、仅参数绑定不同）。对复杂 SQL，每次都优化（Opt-Always）消耗大量 CPU/内存；另一种极端是只优化第一个实例并缓存其计划（Opt-Once），虽最小化优化时间，但缓存计划对后续实例可能任意次优。PQO 是折中：用较少缓存计划，把执行次优性压到尽量低。任一 PQO 技术须为某参数化查询识别一组待缓存计划（populateCache），并在新实例到来时从中高效挑选最佳计划（getPlan）。getPlan 的次优性 = getPlan 所选计划的优化器估计代价 / 缓存中最佳计划的优化器估计代价；整体次优性 = 两者之积。由于 getPlan 在关键路径上，效率（getPlan 耗时）同样关键。

先前 PQO 多为在线技术，其 getPlan 常在对「代价如何随选择性多维空间变化」的假设（如单调性）上失败，导致次优性很高；或采用更保守假设而需频繁回退优化器/重算，导致高均值与尾延迟。本文提出：架构上把 populateCache 与 getPlan 解耦——getPlan 只从缓存中选、不回退，立刻解决延迟；populateCache 离线运行，更可预测、可调试。

2 日志驱动的 PQO 架构

现代 DBMS 普遍具备记录查询实例的日志能力（含参数绑定、执行时间、CPU 时间）。populateCache 模块以 workload 实例集与计划预算 K 为输入，通过 optimize 与 recost 两个 API 咨询优化器，确定要缓存的计划集并安装进计划缓存。训练 getPlan 模型的离线模块以计划集 𝒫 与日志为输入，通过 recost 接口得到（计划,代价）矩阵作为标签，训练出对该参数化查询专用的 ML 模型。在线 getPlan 解析新实例、匹配模板、抽取各谓词选择性作为 ML 模型输入，返回所选计划。

📌 图 2（论文原图）：给定参数化查询的 ML 模型设计——分类（多类，直接标最佳计划 id）与回归（每计划一个回归器预测代价、取最小）两种形式。

3 基于优化器代价的 getPlan

为每个缓存计划计算代价再取最小固然准确，但需要 recost 调用（对复杂查询每计划至少 10ms、常达数百 ms），不实用。我们改用监督 ML：用选择性向量作特征（约 25µs/选择性即可获得），训练极小内存、极快推理的模型。分类形式把问题看作 K 类分类，但「最大化分类准确率」与次优性指标不对齐——错分代价取决于具体计划与实例（样例依赖代价敏感分类），多类难处理（论文 Figure 3 举例）。回归形式为每个计划各训练一个回归器预测其优化器代价，取最小者，是对 recost 的低开销近似，且拥有全部计划的代价信息，表现更好。

4 基于执行代价的 getPlan

优化器代价最低的计划，其执行代价未必最低（基数估计误差、代价模型缺陷）。把问题表述为情境多臂老虎机（CMAB）：臂 = 缓存计划，上下文 = 选择性（或参数绑定）向量，奖励 = 所选计划的（负）执行代价。我们采用在线 Bootstrapped Thompson Sampling：每计划维护 C 个回归模型，对新实例各随机抽 1 个模型预测代价、选最小者执行，再用真实执行代价以 Gamma(1,1) 加权更新这些模型。初期随机决策较差，可先以优化器代价基线（或前 50 例并行执行所有计划取最先完成者）兜底，再切换到老虎机。

5 populateCache 算法

分三阶段：计划收集（对子集实例调优化器）、计划重算（对每个候选计划在子集实例上 recost 得 plan-recost 矩阵）、K-set 识别（从该矩阵确定预算 K 下的计划集，对最大代价指标是 NP 完全）。K-set 识别用简单贪心（从空集起每轮加入使聚合次优性最小的计划，200 计划/2000 实例/K=10 下 <1 秒）。计划收集随机选 100–200 实例、重算随机选 200–400 实例即可取得好结果。数据/workload 显著变化时需重跑离线步骤；老虎机则可持续自适应。

6 实验（节选）

在 SQL Server 2017 上用 TPC-DS、TPC-H skewed、真实库 REAL1，共 25 个复杂参数化查询，每模板 2000 实例，默认 XGBoost（16KB）、选择性特征。结果：(1) 基于优化器代价的 getPlan：监督 ML 在次优性与 getPlan 时间上均显著优于 Opt-Once、MR、PCM、SCR、Density，且方差最低（图 5/6）；100 训练例下分类尾次优性可达 100、回归约 5%（图 7）。(2) populateCache：K=6 时几何均值次优性 <1.2，100-200 优化 / 200-400 重算实例足够（图 8）。(3) 端到端（图 9）：回归模型训练量 −=100, .=200 即合理，初始优化器开销很快摊薄。(4) 执行代价 getPlan（图 10/11）：老虎机收敛到比优化器代价基线更低的执行次优性，且能适应 workload 漂移、计划集增减与奖励噪声；TS 优于 ε-greedy 与 UCB。

7 结论

我们重访 PQO 这一重要问题并提出三点关键思想：第一，主张把「识别缓存哪些计划（populateCache）」与「为某实例从缓存中选哪个计划（getPlan）」解耦；第二，用历史查询日志驱动两者，且对任意复杂 SQL 不做代价行为假设；第三，把 getPlan 拓展为基于执行代价（而非优化器估计代价）的情境老虎机表述，初步实验显示其执行代价低于传统 PQO formulation 下的最佳方案。
