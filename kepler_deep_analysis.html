1研究动机：为何参数化查询优化需要「鲁棒」

参数化查询（parametric query）是指 SQL 结构完全相同、仅绑定的参数值不同的查询，例如同一模板的不同用户 ID。它们在生产中高频重复执行，是数据库优化的「低垂果实」。Kepler 的核心主张是：把优化问题限定在参数化查询这一更可控的设定下，可以用「学习」的方式稳定地超过传统优化器，同时把性能回退（regression）降到最低。

传统学习型优化器（LQO）的四类通病

① 推理开销高：许多学习型方法推理时间比传统方法还慢，难以放进查询关键路径。

② 跨数据分布不稳定：在不同规模、不同分布的数据集上表现忽好忽坏，缺乏一致性。

③ 收益不清晰：很多系统究竟把端到端延迟改善了多少，说不清楚。

④ 缺乏鲁棒性（最关键）：在生产中，任何一次「比原有优化器更慢」的回退都是不可接受的，而学习方法通常无法保证自己的每一次预测都更快。

Kepler 的三个关键差异（相对此前工作 [34] 的 decouple 思路）：
1. 更好的候选生成：提出 Row Count Evolution（RCE），通过扰动优化器的基数（cardinality）估计来高效生成候选计划，往往能找到比内置优化器快得多的计划——甚至比「精确基数（exact cardinality）」计划还快。
2. 用真实执行数据训练：不再依赖容易失准的代价模型（cost model），而是真正执行候选计划、用实际延迟做监督信号，规避了「代价估计 ≠ 实际延迟」的经典错位。
3. 鲁棒模型兜底：用 Spectral-normalized Neural Gaussian Processes（SNGP）量化预测置信度，不确信时直接回退到数据库自带优化器，大幅减少尾延迟与回退频率。

一句话总结贡献

Kepler 首次给出「针对参数化查询、能稳定超过经典优化器」的实用范式；提出 RCE 候选生成算法；证明鲁棒 ML 能在捕获大部分加速的同时极大降低回退风险；端到端集成到 PostgreSQL 且推理开销可忽略；并开源了约 14.2 CPU 年的查询执行数据集（首个面向 PQO 的基准）。

2框架概览：解耦生成与预测 + Trainer/Client 架构

Kepler 沿用了 [34] 的「解耦」思想：把问题拆成两个独立的子问题——(1) 候选生成（candidate generation）为模板 𝑄 生成一组候选计划；(2) 最优计划预测（best-plan prediction）学习一个从查询实例到计划的映射 𝑀。与 [34] 想「匹配内置优化器」不同，Kepler 的目标是尽可能超过内置优化器。

2.1 问题设定（3.1）

模板 𝑄 含 𝑚 个参数化谓词 𝑥₀…𝑥ₘ₋₁。一个查询实例 𝑞 是 𝑄 绑定了具体参数值后的实例。计划 𝑝 描述如何执行任意 𝑞∼𝑄。子计划（sub-plan）是 𝑄 只保留部分表的结果，其输出行数称为子计划基数。默认计划 𝑝_default(𝑞) 是内置优化器为 𝑞 选的计划。

图 2（Kepler 架构）：左侧 Trainer 从生产日志聚合模板、用 RCE 生成候选、执行收集延迟、训练模型并部署；右侧 Client 在收到查询时做推理，仅当置信度高于阈值才下发 plan hints，否则回退优化器。

2.2 Trainer 职责

从生产 DBMS 的查询日志中聚合出查询模板；对每个模板 𝑄ᵢ 用 RCE 生成候选计划 𝑝_k 并执行、收集执行统计；用真实延迟训练一个「参数值 → 最优计划」的模型；把训练好的模型部署回生产库。

2.3 Client 职责与兜底

维护「模板 → 模型」映射。收到 𝑞 时先做模型推理预测最佳 plan hints，仅当置信度分数高于阈值时才把 hints 交给优化器；否则回退到内置优化器。这样即便模型不在分布内（OOD），也不会产生灾难性回退。

2.4 关键设计选择（3.3）

用真实执行延迟：目标就是端到端延迟，因此必须在真实库上执行以获得真值；可用临时（ephemeral）实例隔离训练负载。
减少对代价模型的依赖：图 3 显示 PostgreSQL 估计基数中 64% 都估成 1（独立性假设所致），因此 Kepler 不靠代价估计评判计划好坏。
回退优化器：不确信时就回退，推理开销极低，整体几乎总是低于「永远优化（Opt-Always）」。
模板独立：每个模板各自生成候选、收集数据、训练专属模型，避免互相拖累、便于隔离回退。

3问题定义：候选集、最优加速比与尾延迟

Kepler 把参数化查询优化形式化为两个可度量的子问题，并定义了用来评估「生成」与「预测」各自贡献的指标。

3.1 两个子问题

(1) 候选生成：从指数级大的所有可能计划空间 𝒫 中，为模板 𝑄 生成候选集 {𝑝₀,…,𝑝_{𝑘−1}}（对应 [34] 的 populateCache）。

(2) 最优计划预测：学习映射 𝑀:𝒲→𝒫，最小化执行延迟类目标（对应 [34] 的 getPlan）。

Sₒₚₜ(𝒫,𝒲) = Σ_{𝑞∈𝒲} ExecTime(p_default, 𝑞) / Σ_{𝑞∈𝒲} ExecTime(p*ₒₚₜ, 𝑞)  —  公式(1) 神谕加速比

其中 p*ₒₚₜ 是候选集 𝒫 上的最优计划（实际执行时间最小）。该比值即「若有神谕告知每实例最优计划，总执行时间可缩小的倍数」。由于可把默认计划并入 𝒫，该比值下界为 1。

S_model(𝒲) = Σ_{𝑞∈𝒲} ExecTime(p_default, 𝑞) / Σ_{𝑞∈𝒲} ExecTime(p_model, 𝑞)  —  公式(2) 模型加速比

模型加速比衡量「模型执行整个 workload 比默认优化器快多少」。它上界是 Sₒₚₜ，但下界不是 1——模型完全可能选到比默认还差的计划。二者的关系为 S_model = p · S_RCE，其中 0≤p≤1 是模型捕获到的加速比例。

端到端指标

S_model：模型最终带来的整体加速。

生成指标

S_RCE：候选集本身包含多快计划（神谕上限）。

模型指标

p（捕获比例）与 P_reg（回退频率，模型比默认慢 ≥10% 的比例）。

尾延迟目标：Kepler 还额外关注 workload 的尾延迟。论文定义 99 分位加速比 P99_method(𝒲) = p99({ExecTime(p_default,𝑞)}) / p99({ExecTime(p_method,𝑞)})，对「要求更均匀运行时间」的场景很重要。

4核心算法 Row Count Evolution（RCE）与实例

候选生成的目标是在「包含近最优计划」与「候选集足够小以便训练执行」之间取得平衡。RCE 的核心洞见是：基数误估计是优化器次优的根本原因。因此 RCE 不去直接找单个好计划，而是去「覆盖」一个含近最优计划的集合——它通过随机扰动优化器的子计划基数估计，让优化器重新规划，从而产生多样化候选。

4.1 RCE 算法（Algorithm 1）直觉

RCE 是一种进化式（evolutionary）算法：维护若干「代（generation）」计划。第 0 代只有优化器原本的计划。对后续每一代：先从父代均匀采样若干 base 计划，对每个 base 计划只扰动它实际出现的那些子计划的 join 基数（乘以指数间距的因子），把扰动后的基数喂回优化器得到子代计划；只保留「从未出现过的」计划进入下一代。如此迭代 𝐺 代。

图 4（论文 Figure 4 示意）：RCE 用指数间距的基数扰动（base=10, range=1 → [4,40,400] 与 [1,17,170]）喂给优化器，逐代演化出 C₁、C₂ 计划集。

4.2 论文给出的具体例子（第 9–10 页）

某查询实例有两个 join：A⋈B 估计基数 40，C⋈D 估计基数 17。RCE 用 base=10、range=1 构造候选行数集合：A⋈B → [4, 40, 400]，C⋈D → [1, 17, 170]。

① RCE 从这些集合中均匀采样新 join 基数；例如一次采样为 400 和 17，就「诱导」优化器产生一个新计划 Plan-1（Gen 1）。重复 N 次得到 C₁ 个 Gen-1 计划。

② 接着从 Gen-1 去重后的计划里采样 S 个，对每个再随机扰动 N 次，生成 Gen-2 的 C₂ 个计划。

③ 最终候选集 = C₀ ∪ C₁ ∪ … ∪ C_G。

4.3 RCE 的几种解读

精确基数匹配：RCE 高效构造了一个「覆盖多种精确基数」的计划集合；扰动足够多时，任一实例的精确基数大概率是其中某个扰动点的近邻，对应计划也相近。

乘法扰动：符合基数估计的 Q-error 度量；指数间距的扰动集与优化器 Q-error 分布有相似支撑。

局部搜索：RCE 本质是在「子计划基数」这个低维子空间里做随机游走，起点是优化器估计（虽不准但比随机更有信息）。

4.4 为什么只扰动「相关」子计划

一个 n 表 join 共有 2ⁿ−1 个子计划，但 RCE 只扰动采样计划中实际出现的 n−1 个子计划的基数。这极大提升效率、仅损失很小 generality：因为扰动在代际间继承，只有「基数被严重高估」的子计划才永不被扰动——而优化器因独立性假设往往低估而非高估，故该情况罕见。

5训练数据采集：自适应超时与计划覆盖剪枝

生成候选集 𝒫 后，需要在训练 workload 上执行每个计划，得到「实际执行延迟」数据集，用于监督式最优计划预测。Kepler 用两个关键技术降低昂贵的执行成本。

5.1 执行机制

通过 hint 强制优化器产出指定 join/scan 方法与 join 顺序的计划；在多个库中并行执行查询实例与其候选计划；为模拟 warm buffer cache，每个计划执行多次取最小值作为估计延迟。重复执行也降低了测量噪声。

5.2 自适应超时 + 计划重排

借鉴 [37] 的超时策略并做两处改进：① 总是先执行默认计划，再按「历史执行延迟升序」重排其余计划——这样超时阈值能尽快收紧，把资源留给可能更快的计划；② 每个计划的第一次执行不适用收紧后的超时，以保证 warm-cache 模拟。

5.3 在线计划覆盖剪枝（Plan Cover Pruning, PCP）

先用前 N 个查询实例执行全部计划，再用集合覆盖（Set Cover）把一个模板的数百个候选剪枝到最小的「计划覆盖集」：

· 若某计划对实例 𝑞 的执行时间落在「目前所见最快时间的 1+ε 倍以内」，称其为该实例的近最优计划；每个计划对应一组它近最优的实例。

· 计划覆盖集 = 能覆盖每个实例至少一个近最优计划的最小计划集合。用标准贪心近似：每轮选「能覆盖最多剩余实例」的计划。

· 放宽到只需覆盖 1−δ 比例的实例，在「覆盖集大小」与「可达加速」间权衡。模型之后只从这少数计划里预测。

尾延迟重排：很多模板默认延迟呈重尾分布，尾部的参数往往更次优、对总加速影响更大。Kepler 先评估这些尾部实例，使覆盖计算包含它们——这一重排在整个 Stack 数据集上带来 7–8 倍的总执行时间与计划数减少。

6鲁棒最优计划预测：特征、多标签损失与 SNGP 兜底

收集到延迟数据集后，Kepler 为每个模板训练一个模型，目标是最大化加速比同时最小化回退；并在置信度低时回退到优化器。本节拆解特征、训练目标与模型三类设计。

6.1 特征：只用原始参数值

Kepler 仅用模板的 𝑚 个参数值作为输入特征（数值归一化到 N(0,1)、字符串做 embedding、日期转数值），不转换成基表选择性（selectivity）。原因：① 选择性是信息有损表示，两个不同值可能有相同选择性；② 当优化器基数估计次优时，选择性更差（见图 3）。

字符串词表选择：为每个字符串列建固定大小词表（one-hot + embedding，含 OOV 桶）。词表用最大边际改进（max marginal improvement）策略选取——优先纳入「能带来最大总改进」的 top-k 值：

m(v, i) = Σ_{𝑞∈𝒲, 𝑥ᵢ=v} [ ExecTime(p_default, 𝑞) − ExecTime(p*ₒₚₜ, 𝑞) ]  —  公式(3)

6.2 训练目标：多标签分类 + 样例依赖损失

多标签分类：把最优计划预测建模为多标签分类——每个近最优计划都给正标签（而非只标唯一最优）。这提供了更丰富的监督信号，改善中间表示质量。用「单标签化」的二分类交叉熵训练每个候选计划的近最优概率。

近最优定义：计划 𝑝 近最优当且仅当 (ℓ_d − ℓ_p)(1+τ) ≥ (ℓ_d − ℓ_o)，其中 ℓ_d 为默认计划延迟、ℓ_p 为 𝑝 的延迟、ℓ_o 为最优延迟，τ>0。

为何不用回归/多类：回归跨大范围不稳定、且被超时掩盖了次优计划的真实延迟，还要求了不必要细粒度；单标签多类在「多个计划都近最优、延迟有噪声」时表现差（两个计划概率相等→低置信）。

样例依赖损失：不同实例对总体目标的贡献不同，用样本加权：比默认差的计划权重 ×C；近最优计划按改进量软加权 1 + D·log(ℓ_d − ℓ_p)。

图 6（模型结构）：共享表示 + 每计划一个输出头（提升推理速度、缩小模型），头接 SNGP 给出校准概率与不确定性，低置信时回退优化器。

6.3 模型与不确定性（SNGP）

基础模型用简单前馈神经网络（3 层 64 隐单元，Adam lr=3e-4，ReLU）。为推理效率，采用「一个共享表示 + 每计划一个输出头」的结构，优于「每计划一个独立模型」。

不确定性：Kepler 需要校准的预测与不确定性估计以避免选到明显次优计划。两种主流方案：集成（ensemble，成本线性 ×M）与 SNGP。Kepler 选 SNGP——对所有层做谱归一化（提供 bi-Lipschitz 保证），并用高斯过程输出层高效估计不确定性，开销更低。

兜底：预测置信度低于阈值（默认 0.9）时回退到默认计划。

10总结、局限与未来工作

Kepler 用「RCE 生成更好候选 + 真实执行数据训练 + 鲁棒 SNGP 兜底」三件套，在 PostgreSQL 上稳定、显著地超过内置优化器，且把回退降到可忽略。它首次证明：在参数化查询这一可控设定下，学习型优化器可以既快又稳。

核心结论：① RCE 候选生成能带来显著执行加速，且常优于「精确基数」计划；② 鲁棒 ML 模型能在捕获大部分加速的同时极大降低回退；③ 端到端 PostgreSQL 集成推理开销可忽略；④ 发布的 14.2 CPU 年数据集可推动后续 PQO 研究。

局限

Kepler 面向「执行频繁到足以摊薄训练采集成本」的参数化查询；当前假设系统状态（配置、优化器、数据分布）相对固定，变化缓慢时可周期性重采重训；训练数据采集本身较昂贵（虽可大幅优化）。

未来方向

· 用生成式基数分布（如 NeuroCard）做更高概率的扰动采样，而非均匀扰动；
· 把查询计划树与选择性估计作为模型额外特征，让模型自行判断基数何时可靠；
· 在模板间共享结构信息而不增加回退风险；
· 用主动学习降低训练数据采集成本；
· 扩展到 PostgreSQL 之外的 DBMS（论文认为其经验式本质可跨库发现收益）。

Q1：Kepler 和经典 PQO（如 [34]）最大的区别是什么？
[34] 的 PQO 目标偏「减少规划次数、匹配内置优化器」；Kepler 目标偏「尽可能超过内置优化器」并强调鲁棒性。它因此用 RCE 主动找更快计划、用真实执行数据而非代价模型、用 SNGP 兜底避免回退。
Q2：为什么 RCE 比「精确基数（EC）计划」还好？
EC 计划只保证基数精确，但代价模型还有独立性假设、启发式等其他缺陷，仍可能选次优计划。RCE 直接在「实际延迟」空间探索，绕过了代价模型的整套假设。
Q3：SNGP 的「兜底」到底解决什么问题？
在生产中，任何一次比默认优化器更慢的预测都不可接受。SNGP 给出校准的不确定性；当模型对输入不确信（如 OOD）时回退到优化器，把回退频率从最高 23.8% 降到 0.1%~0.2%，且仅牺牲少量加速。
Q4：Kepler 适合我的业务吗？
如果你的查询是参数化、且同一模板被高频重复执行（如多租户 SaaS 的固定报表查询），Kepler 的「训练采集成本」能被充分摊薄，收益显著；若查询一次性、低频，则成本不划算。

标题与摘要

Kepler：面向更快参数化查询优化的鲁棒学习。大多数既有参数化查询优化（PQO）技术依赖传统查询优化器的代价模型，而这些模型常常不准，导致次优的查询性能。我们提出 Kepler，一种端到端、基于学习的 PQO 方法，相比传统查询优化器在查询延迟上取得了显著加速。我们方法的核心是行数演化（Row Count Evolution, RCE）——一种基于子计划基数空间扰动的新型计划生成算法。先前的方法需要准确的代价模型，而我们通过「用真实执行数据评估候选计划、训练一个 ML 模型来预测给定参数绑定下最快的计划」绕过了这一要求。我们的模型利用神经网络不确定性领域的最新进展，鲁棒地预测更快计划，同时避免查询性能的回退。实验上，我们展示 Kepler 在多个 PostgreSQL 数据集上取得了查询运行时间的显著改进。

📌 图 1（论文原图）：Kepler 在 Stack 上取得整体 2.41× 加速，来自三部分的合力：① 通过 RCE 发现更好的计划；② 用 SNGP 模型捕获大部分加速同时最小化回退；③ 模型推理时间极快。

1 引言

参数化查询优化（PQO）旨在优化参数化查询，即 SQL 结构相同、仅绑定参数值不同的查询。这类查询在现代数据库使用中无处不在，并且因为被反复执行，存在巨大的性能提升机会。然而，PQO 主要从「减少规划时间、尽量避免重新优化」的角度被研究，这类方法隐式受限于系统查询优化器的性能，从而继承了传统优化器所有被充分研究的次优性。理想的参数化查询系统不仅应通过 PQO 最小化规划时间，还应通过查询优化（QO）优化执行性能。

多种方法尝试用机器学习改进查询优化，但不幸的是，大多数学习型技术至少有四个缺陷：(1) 推理时间高于传统方法；(2) 在不同数据集规模与分布上表现不一致；(3) 查询性能改进往往不清晰；(4) 缺乏鲁棒性——在生产中，性能回退是不可接受的，而这对学习方法尤其困难。

我们提出，把查询优化问题限定在参数化查询设定下，是一个更可学习的、因而能被更鲁棒地解决的问题。为此我们提出 Kepler（K-plan Evolution for Parametric Query Optimization: Learned, Empirical, Robust），一种面向参数化查询的端到端学习方法。它建立在 PQO 既有工作 [34] 之上，沿用「解耦的计划生成 + 基于学习的计划预测」架构，但有三点关键差异：第一，Kepler 提出 RCE，通过扰动优化器的基数估计高效生成候选计划，仅需任意标准代价优化器的简单接口；第二，Kepler 用真实查询执行数据构建训练数据集，规避了代价模型与实际延迟之间的错位；第三，Kepler 用鲁棒的神经网络预测技术（SNGP）降低尾延迟与查询回退，不确信时回退到数据库优化器。

📌 图 2（论文原图）：Kepler 架构，分为 Trainer（从生产日志聚合模板、RCE 生成候选、执行收集统计、训练并部署模型）与 Client（收到查询实例时做推理，仅当置信度高于阈值才下发 plan hints，否则回退优化器）。

3 概览（节选）

Kepler 的问题设定：模板 𝑄 含 𝑚 个参数化谓词；查询实例 𝑞 是绑定了具体值的实例；子计划是 𝑄 只保留部分表的结果，其输出行数称为子计划基数；默认计划 p_default(𝑞) 是优化器为 𝑞 选的计划。Kepler 把问题解耦为：(1) 候选生成——为 𝑄 从指数级大的计划空间生成候选集；(2) 最优计划预测——学习映射 𝑀:𝒲→𝒫 最小化执行延迟。

我们定义神谕加速比 Sₒₚₜ(𝒫,𝒲)（公式 1）为默认计划总执行时间除以候选集上最优计划总执行时间，它等价于 [34] 中执行代价次优性的定义，重命名为「加速比」以强调目标差异。模型加速比 S_model(𝒲)（公式 2）衡量模型执行 workload 比默认快多少，其上界为 Sₒₚₜ、但下界不是 1。二者的关系为 S_model = p·S_RCE。

4 行数演化（RCE）

RCE 的核心想法是：基数误估计是优化器次优的根本原因；而候选生成阶段只需要「生成一个包含（近）最优计划的集合」，而非直接识别单个好计划。RCE 以进化式算法实现：维护若干代计划，第 0 代只有优化器原本的计划；对后续每代，从父代均匀采样 base 计划，对每个只扰动其实际出现的子计划 join 基数（乘以指数间距的因子），把扰动后的基数喂回优化器得到子代，只保留未出现过的计划进入下一代。RCE 仅扰动采样计划中实际出现的 n−1 个子计划（而非全部 2ⁿ−1 个），大幅提效且损失很小。

论文举例：某实例 A⋈B 估计基数 40、C⋈D 估计基数 17；用 base=10、range=1 构造候选行数集合 A⋈B→[4,40,400]、C⋈D→[1,17,170]。一次采样（400, 17）诱导优化器产生 Gen-1 的 Plan-1；重复 N 次得 C₁；再从 C₁ 去重采样 S 个、各扰动 N 次得 C₂。最终候选集 = C₀∪C₁∪…∪C_G。

5 训练数据采集

生成候选集后，在每个计划上执行以收集实际延迟。Kepler 用 hint 强制计划、并行执行、取多次最小延迟作为真值；用自适应超时（先执行默认计划、按历史延迟升序重排其余计划以尽快收紧超时）与在线计划覆盖剪枝（PCP，用贪心集合覆盖把数百候选剪到最小覆盖集，仅需覆盖 1−δ 比例实例）降低成本。尾延迟重排使 Stack 上总执行时间与计划数减少 7–8×。

6 鲁棒最优计划预测

特征上仅用原始参数值（不转选择性），字符串列用最大边际改进策略选词表（公式 3）。训练目标为多标签分类（每个近最优计划都标正），并用量例依赖损失加权。模型用前馈神经网络 + 每计划一个输出头；不确定性用 SNGP（谱归一化 + 高斯过程输出层），置信度低于 0.9 时回退优化器。

7 实验（节选）

在 PostgreSQL 13.5 上，Stack 取得整体 2.41× 几何均值加速，32.2% 的查询超过 2×。RCE 在 78/87 个 Stack 查询上超过 1.2×；模型在 80%+ 查询上捕获 80%+ 的 S_RCE。与 Bao 对比（表 2），Kepler 在 6 个模板中 5 个胜出，q15_0、q16_0 上远超前者。表 5 显示 RCE 计划在 6 个查询中 5 个比精确基数计划更快。图 14 的 OOD 实验表明，开启兜底后回退频率从最高 23.8% 降到 0.1%~0.2%。

8 结论与未来工作

我们提出 Kepler，一种能用基于学习的方法鲁棒加速参数化查询的系统。我们广泛评估表明：(1) 新型候选生成算法 RCE 能带来显著执行延迟加速；(2) 鲁棒 ML 模型能可靠预测更快计划同时避免回退；有趣的是，RCE 生成的计划常常远优于精确基数计划，说明即便 PostgreSQL 这样广泛使用的系统也有巨大的改进空间。未来方向包括：用生成式基数分布（如 NeuroCard）做更高概率扰动、把计划树与选择性作为模型特征、在模板间共享结构、用主动学习降低训练采集成本，以及扩展到 PostgreSQL 之外的 DBMS。

关于训练成本

论文为得出更稳健的结论并产出完整数据集，采用了昂贵的训练数据采集流程（平均每模板约 39 CPU 天）。对实际应用，可通过主动学习等手段大幅降低该成本；结合图 16，相似性能可用显著更少的训练成本获得。
