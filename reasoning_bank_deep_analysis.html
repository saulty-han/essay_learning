1 研究动机：持久化智能体无法从经验学习

持久化 LLM 智能体持续遇到任务流，但无法从累积经验学习：会重复过去错误、丢弃有用洞察、缺乏自演化能力。现有记忆方法只存原始轨迹（trajectory memory）或仅成功例程（workflow memory），有两个缺陷：① 不能提炼可迁移的高层推理模式；② 过度强调成功、忽视失败教训。

原始轨迹 / 成功流程的不足

trajectory memory 存原始交互，难以复用；workflow memory 只存成功流程，无法从失败中获益。两者都缺"提炼通用推理策略"的能力。

失败的价值被忽视

仅从成功学习会错过失败中的反事实信号与陷阱。ReasoningBank 主张同时从成功与失败中提炼可泛化的推理策略。

本文主张：推理记忆

把经验蒸馏为带 title / description / content 的结构化记忆项（memory items），并用 MaTTS（memory-aware test-time scaling）建立记忆与测试时扩展的双向协同，实现智能体的自我进化。

2 框架概览：记忆蒸馏闭环

ReasoningBank 把经验蒸馏为结构化记忆项，闭环运行：新任务 → 检索相关记忆 → 指导交互 → 提取新记忆 → 整合回 Bank。智能体策略 π_L(·|M,A) 以 ReasoningBank M 为额外系统指令。

测试时学习（streaming）

任务 q1→q2→…→qN 顺序到达无 GT。每个任务：检索 top-k 记忆 → 注入系统指令 → ReAct 式交互 → judge 标注 → 提取成功 / 失败记忆项 → consolidation 加法并入 Bank。MaTTS 时生成 k 条轨迹，parallel 用 self-contrast 筛选、sequential 用 self-refinement 丰富记忆。

3 推理记忆的提取与组织

记忆项三字段：title（标题）、description（一句话描述）、content（提炼的推理步骤 / 决策依据）。集成三步：检索、提取、整合。

三阶段集成

(i) Memory Retrieval——用当前查询上下文做 embedding 相似度检索 top-k 记忆项注入系统指令；(ii) Memory Extraction——用 LLM-as-a-Judge 无 GT 标注判定轨迹成功 / 失败，成功贡献已验证策略、失败提供反事实信号 / 陷阱；(iii) Memory Consolidation——以简单加法操作并入 ReasoningBank，维持演化仓库。

4 MaTTS：记忆与测试时扩展协同

LLM-as-a-Judge 在测试时无 GT 下为轨迹提供代理正确性信号（success/failure），judge 准确率约 72.7%（baseline），且在 50%–100% 准确率下性能稳定。

Parallel Scaling

同一查询生成 k 条轨迹，跨轨迹 self-contrast 对比筛选出一致模式、过滤虚假解，Best-of-N 选最终。利用同题冗余探索的对比信号做记忆筛选。

Sequential Scaling

单轨迹内 self-refinement 迭代精炼，中间笔记也作为记忆信号。缩放因子 k = 并行轨迹数 / 串行精炼步数。

双向协同

高质量记忆引导扩展走向更有希望的路径（BoN 从 39.0 升到 55.1），而多样轨迹又反哺筛选出更高质量记忆，形成正循环。

5 自我进化循环

失败利用的实证

仅成功时 ReasoningBank 46.5，加入失败达 49.7；而 Synapse 40.6→41.7、AWM 44.4→42.2（下降）——证明失败信号对 ReasoningBank 是增益、对仅存成功的方法则无益。

emergent behavior：记忆项从执行策略 → 自适应自反思 → 自适应检查 → 组合策略演化，体现记忆随任务流不断精炼。

6 记忆与进化的整合

端到端数据流

每个任务：检索 top-k 记忆 → 注入系统指令 → ReAct 交互 → judge 标注 → 提取成功 / 失败记忆项 → consolidation 加法并入 Bank。MaTTS 时：在检索记忆引导下生成 k 条轨迹，parallel 用 self-contrast 筛选可靠记忆、sequential 用 self-refinement 丰富记忆，形成"高质量记忆引导扩展、丰富扩展反哺记忆"的正反馈。

9 代码使用与实战（含 GitHub 源码、安装命令、使用示例）

官方仓库

代码仓库：https://github.com/google-research/reasoning-bank（UIUC & Google Cloud AI）。基于 Gemini-2.5 / Claude-3.7 通过 BrowserGym + ReAct 构建；数据集 WebArena、Mind2Web、SWE-Bench-Verified 均公开。

# 1) 克隆仓库
git clone https://github.com/google-research/reasoning-bank.git
cd reasoning-bank

# 2) 安装依赖（BrowserGym + ReAct 环境）
pip install -r requirements.txt

# 3) 运行 WebArena 自演化（检索 + judge + consolidation）
python run_reasoning_bank.py --benchmark webarena \
    --model gemini-2.5-flash --top_k 5

# 4) 启用 MaTTS（parallel / sequential scaling）
python run_matts.py --mode parallel --k 5 --benchmark webarena

复现要点：Memory Retrieval 用 embedding 相似度；LLM-as-a-Judge 提供无 GT 的成功 / 失败信号；Consolidation 用加法并入；MaTTS 通过 self-contrast / self-refinement 放大收益。

10 总结与未来展望

ReasoningBank 把经验蒸馏为带 title/description/content 的推理记忆，从成功与失败双向学习，并用 MaTTS 建立记忆与测试时扩展的双向协同。在 WebArena / SWE-Bench / Mind2Web 上显著提升效果并降低交互步数。

Q1：ReasoningBank 和普通记忆有何不同？
不只存原始轨迹或成功流程，而是提炼带 title/description/content 的通用推理策略，并从失败中提取反事实教训。
Q2：MaTTS 怎么让扩展更有效？
并行用 self-contrast 跨轨迹筛选一致模式，串行用 self-refinement 把中间笔记转为记忆信号，把冗余探索变成对比信号。
Q3：为什么失败也有用？
LLM-as-a-Judge 标为失败的轨迹提供陷阱 / 反事实信号，蒸馏成 guardrail，使记忆更鲁棒（Synapse/AWM 因只用成功无法受益）。

局限与未来

依赖 LLM-as-a-Judge 质量（虽鲁棒但非完美）；记忆以简单加法整合（可进一步用更精细检索 / 整合）；主要在 web / SWE 两类任务验证。
