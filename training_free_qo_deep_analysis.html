1研究动机：为什么用「计划嵌入」做免训练优化

查询优化的核心困难在于代价模型与基数估计不准，导致优化器选错计划（例如本该用 hash join 却用了 nested-loop）。DBA 可用 hint 引导优化器，但「选对 hint」极其复杂——需要深度理解查询、数据与优化器内部启发式，且错误 hint 反而拖慢。近年来用 ML 自动化这一过程的趋势有两条：① 端到端替换优化器（如 Neo）；② 以 hint/代价修正来引导现有优化器（如 AQO、ACM、Bao、AutoSteer、FASTgres、COOOL、Lero、HERO）。

LLM 用于查询优化的两条路线：
· 生成式（LLM-QO、LLM-R2）：用 LLM 直接「写」执行计划或重写规则。潜力大但推理成本高、延迟大、离实用尚远。
· 调优式（LLMSteer 等）：保留优化器，用 LLM 嵌入查询文本，再训练小分类器选 hint。LLM-PM 更接近这一支，但更进一步——嵌入的是「执行计划」而非「查询文本」，且完全免训练。

本文立场

目标不是追求极致加速，而是证明「执行计划嵌入」的实用价值，并构建一个简单、鲁棒、可用于生产的系统。LLM-PM 刻意轻量、免训练：用冻结（frozen）的预训练 LLM 嵌入计划，再用简单的近邻检索来迁移 hint 集。相比生成式 LLM，仅用嵌入便宜得多、快得多；近邻检索高效、鲁棒、透明，是可插拔、易适应新 workload/模式/数据分布的机制。

为什么「免训练」重要？ 端到端学习型优化器（Neo 等）需要海量标注数据与漫长训练，还丢弃了成熟优化器的数十年手工启发式，且泛化存疑。LLM-PM 把「重活」交给预训练 LLM 的语义表示，规避了训练阶段，契合「快速、高效优化系统」的需求。

2框架概览：Plan-Mapping + 自适应 Hint 搜索双阶段

LLM-PM 由两部分组成：(1) 计划映射算法（Plan-Mapping）——在嵌入空间把相似计划的 hint 集迁移过来；(2) 自适应搜索（Adaptive Hint Search）——穷举探索整个 hint 空间找最优配置作为兜底。搜索算法本身不新颖，但作为兜底保证完备性与系统自洽。

图 2（论文 Figure 2 总体系统控制流）：Plan-Mapping 走快路径；失败则兜底到 Adaptive Hint Search；三元组持续反哺参考集。

2.1 快路径（Plan-Mapping）

用全 hint 启用规划入站查询、嵌入、与参考 workload 用 Algorithm 2 比对；若候选 hint 通过双邻域一致性校验，则带该 hint 直接执行。

2.2 兜底（Adaptive Hint Search）

当 Plan-Mapping 无 hint 或校验拒绝，转 Algorithm 1：穷举（但带超时剪枝）扫描全部 128 个二进制 hint 集，找到最快配置，并缓存最优的 ⟨默认计划, 最优 hint, 最优计划⟩ 三元组供将来使用。

3问题定义与 7 个布尔 Hint

自动 hint 选择被形式化为：给定查询 Q，找到 hint 集 H 使执行时间最小。

H*(Q) = arg min_{H ∈ H(Q)} T(Q, P_H)  —  公式(1)

其中 H(Q) 是 Q 的所有有效 hint 配置空间。实践中 H(Q) 可能极大，因此用两种技术加速搜索。

3.1 两种加速技术

超时阈值（fastest-so-far）：每次计划执行后更新 Tmin ← min(Tmin, T(Q,P))（初始 Tmin = 默认计划耗时）。若某 hinted 计划超过当前 Tmin 则中止其执行并标记为次优——尽早剪掉昂贵计划。

计划缓存（plan caching）：每个不同物理计划连同其测得的延迟一起缓存；当优化器产生已在缓存中的计划时，直接复用存储的延迟而不再执行。在「许多 hint 对所选计划影响很小」时最有效。

考虑的 7 个布尔 hint（1=禁用该算子，0=保持默认启用）
enable_nestloop / enable_hashjoin / enable_mergejoin
enable_seqscan / enable_indexscan / enable_indexonlyscan / enable_bitmapscan
共 2⁷ = 128 种二进制 hint 集

表：本文考虑的 7 个布尔 hint；注意其编码约定「1 表示禁用该算子」。

3.2 等价 hint 集的消歧（Canonical 选择）

同一物理计划可能由多个 hint 配置生成，例如 H1=(0,1,1,1,0,1,1) 与 H2=(0,1,1,1,0,1,0) 仅末位（bitmapscan）不同却产生相同计划。为选唯一「规范」hint 集，取禁用算子最少的配置：bH(P) = argmin_{H: P_H=P} Σ h_i。上例中 H2 更优（少禁用一个）。理由：① 若某 knob 开/关都产生同计划，说明它对效率无影响，保持启用可避免不必要的限制；② 多关算子会压缩优化器搜索空间，可能在不同于训练 workload 的查询/分布上表现差。最宽松的 hint 集在引导到目标计划的同时，保留了优化器尽可能多的内置经验。

4自适应 Hint 搜索（Algorithm 1）

Algorithm 1 在单次遍历中扫描全部 128 个二进制 hint 集，同时不断收紧执行时间阈值。先用全算子启用执行查询，建立初始 Tmin 并播种计划-延迟缓存 C；其余配置分配到 w=8 个工作线程并行。对每个 hint 集，优化器产生物理计划；若计划已在 C 中则直接取延迟，否则在当前 Tmin 超时下执行；超时的计划被提前中止并标记次优；每当更快的执行完成，Tmin 被原子更新，立即收紧所有在途 worker 的超时。

Algorithm 1 Adaptive Hint Search (Timeout Pruning + Plan Cache)
Input: Q; H={H1..H128}; w←8
Output: Tmin; plan–latency cache C
1: Tmin ← execute(Q, default)
2: C ← { plan(Q,default) : Tmin }
3: for all H in H\{default} parallel on w threads:
4:   P ← plan(Q, H)
5:   if P in C: t ← C[P]
6:   else: t ← execute(Q, H, timeout=Tmin)
7:        if t == Timeout: mark H sub-optimal; continue
8:        C[P] ← t
9:   Tmin ← min(Tmin, t)   # atomic
10: return (Tmin, C)

为什么高效

因为慢计划被快速终止、重复计划通过缓存短路，整个过程把大部分算力花在少数有希望的候选上，同时保证对 128 个 hint 空间的全覆盖。这是「兜底」机制的核心：当 Plan-Mapping 无把握时，仍能通过穷举得到最优配置，并把它缓存为三元组。

5Plan-Mapping 算法（Algorithm 2）：邻域投票 + 双邻域校验

核心假设：LLM 的嵌入能为查询计划提供良好表示；嵌入空间中相近的计划倾向于共享物理结构，因而同一 hint 集常在该局部邻域内都有效。离线搜索（§3.2）产出三元组 (d_i, H*_i, o_i)：d_i 是第 i 个默认计划的嵌入，H*_i 其最优 hint 集，o_i 是对应优化后计划的嵌入（若无 hint 加速则 H*_i 为全启用向量且 o_i=d_i）。

图 1（论文 Figure 1 两阶段 hint 选择）：左—在 N 个最近默认计划邻域中投票选候选 hint；右—用 K 个最近优化计划邻域校验收益。

5.1 两阶段测试

① 邻域投票：在 N 个最近默认计划中，选「最常见的非默认 hint 集」作候选 Hcand。假设：结构相似的计划往往有同样的低效点，故对邻居有效的 hint 也可能以类似方式优化新计划。

② 一致性校验：用所选候选 hint 重规划查询，对默认计划与候选计划分别取其 K 个最近邻（均有已知执行时间），比较两邻域的平均运行时。若候选计划邻域的平均运行时更小，则采纳新 hint；否则保留原计划。

Algorithm 2 Plan-Mapping (Two-Neighborhood Consistency)
Input: Q; reference T={(d_i,H*_i,o_i,t_def_i,t_opt_i)}; N, K
Output: Hout or ∅
1: P0 ← plan(Q, default); p0 ← ϕ(P0)
2: N  ← N nearest {d_i} to p0
3: Hcand ← mode{ H*_i | H*_i≠0, (d_i,..)∈N }   # 非默认中最频繁
4: if Hcand undefined: return ∅
5: Pcand ← plan(Q, Hcand); pcand ← ϕ(Pcand)
6: K0    ← K nearest {d_i} to p0;      t0bar ← mean(t_def over K0)
7: Kcand ← K nearest {o_i} to pcand;   tcandbar ← mean(t_opt over Kcand)
8: if tcandbar < t0bar: Hout ← Hcand  else Hout ← ∅
9: return Hout

5.2 超参数

· N：投票阶段邻域大小（实验 N=16）；
· K：一致性校验邻域大小（K=16；小的 K 偏向最近计划，大的 K 增加覆盖与稳定）；
· 距离度量：默认欧氏距离。

两阶段测试在「覆盖」与「安全」间取得平衡：N 邻域投票编译出本地流行 hint 列表并选最频繁者作候选；K 邻域一致性校验防止盲用会把优化器推入次优空间的 hint。

6整体系统设计：快路径 / 兜底 / 持续学习

把嵌入方法与自适应搜索整合为统一管线，旨在：(1) 当经验库存在相似历史计划时，经由 Plan-Mapping 快速优化；(2) 对新查询兜底到彻底 hint 搜索；(3) 通过持续累积执行经验随时间改进。

阶段 1 · Plan-Mapping

全 hint 启用规划、嵌入、与参考集比对（Alg.2）；候选通过双邻域校验则直接带 Hcand 执行（快路径）。

阶段 2 · 兜底搜索

Plan-Mapping 无 hint 或校验拒绝时，转 Alg.1 穷举 128 hint 集，找到最快配置并缓存三元组。

阶段 3 · 执行与持续学习

兜底产出的三元组不断扩充参考集；随反馈循环，需要慢速穷举的查询比例逐步缩小，端到端延迟稳步改善。

设计收益：低中位延迟（多数查询由轻量嵌入映射器解决），同时保证即使是「困难」查询最终也能从彻底搜索中受益。计划嵌入存储（embeddings storage）是连接两阶段的经验库。

平台差异的警示

先前 PostgreSQL 研究报告：单个「全局禁用 nested-loop」hint 就能让 JOB-CEB 加速 60%+；但在本实验的 openGauss 上，同样的 hint 反而让性能退化约 50%。这正说明「没有哪个 hint 是普适最优」，也是本文采用「多邻域策略」的动机。

一个自然的问题：学到的 LLM 嵌入是否真比「昂贵的字符串匹配」更有用？为此作者关掉一致性校验，用两种退化配置对比：① Levenshtein 编辑距离；② 仅 LLM 嵌入（无一致性校验）。

8.1 结论与未来工作

消融证实：学到的嵌入捕捉到了远超原始字符串相似度的信息，且与一致性校验结合是稳健收益的关键。即便最先进的 OpenAI 嵌入，其训练语料也极少同时包含「查询计划 + hint 如何影响计划」的显式信息——自然下一步是开发/微调领域专用嵌入（含丰富 SQL 语法、优化技术、hint 标注与代价反馈）。

当前 Plan-Mapping 平均约 20% 加速（约为理论上界的三分之一）。剩余差距约一半来自候选 hint 选择阶段：在「oracle」配置下（一致性校验可查看默认与候选计划的真实运行时），可收回近 40% 的可达加速（即最大值的约三分之二）——说明更精细的相似度度量或投票机制有望填补大部分残差。

10总结、局限与未来工作

LLM-PM 是一个轻量、免训练的框架：用预训练 LLM 对执行计划做嵌入，通过 hint 推荐来引导传统代价优化器。它先在 N 个最近默认计划中找出本地流行的 hint 集，再用「比较默认与优化计划区域平均运行时」的双邻域一致性校验验证收益。无需改动底层 DBMS、无需大量离线训练，只复用嵌入空间中编码的历史计划结果。

核心贡献：· 用 LLM 生成的执行计划嵌入增强 hint 选择；· 轻量、快速、免训练，无需大量训练或系统改动即可引导优化器；· 在 JOB-CEB 上证明 LLM 计划 hinting 能稳定超越默认优化器（平均 21.1% 总加速）；· 凸显 LLM 嵌入改善查询性能的潜力，为高效 LLM 驱动优化系统指明方向。

局限与未来

· 当前平均加速约为理论上界的三分之一，候选选择阶段是主要残差来源；
· 通用 LLM 嵌入缺乏「计划–hint–代价」领域语料，领域专用/微调嵌入是下一步；
· 可用度量学习或「计划+SQL 混合特征」精炼嵌入空间，并按局部密度动态适配邻域大小；
· 硬件/引擎异构使跨系统直接对比困难（同 hint 在不同 DBMS 上效果可能相反）。

Q1：LLM-PM 和 LLM-QO、LLMSteer 有何本质区别？
LLM-QO 用生成式 LLM「写」计划（端到端替换优化器，贵且慢）；LLMSteer 嵌入「查询文本」再训小分类器。LLM-PM 嵌入的是「执行计划」本身、且完全免训练，用近邻检索迁移 hint，最贴近生产实用性。
Q2：为什么需要「双邻域一致性校验」？
单靠邻域投票可能盲用会把优化器推入次优空间的 hint。校验通过比较「候选计划邻域的平均优化运行时」与「默认计划邻域的平均默认运行时」，只对确有收益的 hint 采纳，把超时从 5 次降到 1 次、加速从 16.1% 提到 21.1%。
Q3：消融里 Levenshtein 为什么反而变慢 64%？
编辑距离只看计划字符串的表面相似，捕捉不到「结构/语义相近 → 同样 hint 有效」的信息；近邻检索到的「相似」计划实际并不共享低效点，导致推荐错误 hint、大量执行超时，整体严重退化。
Q4：兜底搜索会不会很慢？
单次兜底需扫 128 个 hint 集，但靠超时剪枝与计划缓存把算力集中在少数有希望的候选上；且它是「一次性」地为新查询建立三元组，之后同类查询走快路径即可，随持续学习兜底比例越来越小。

摘要

大语言模型（LLM）嵌入为数据库查询优化提供了一条有前景的新途径。本文探讨预训练的执行计划嵌入如何在不需额外模型训练的情况下指导 SQL 查询执行。我们提出 LLM-PM（LLM-based Plan Mapping）框架：嵌入查询的默认执行计划，在先前执行过的计划中找其 k 近邻，并基于邻域投票推荐数据库 hint 集。一个轻量一致性校验验证所选 hint，而兜底机制在需要时搜索整个 hint 空间。在 openGauss 的 JOB-CEB 基准上评测，LLM-PM 平均实现 21% 的查询延迟缩减。本工作凸显了 LLM 驱动的嵌入带来实际查询性能改进的可能，并为免训练、基于嵌入的优化器引导系统开辟了新方向。

1 引言（节选）

查询优化是关系数据库系统的核心问题：优化器必须从庞大的候选空间（可能的连接顺序、连接算法、表访问方法等）中为每条 SQL 构造高效执行计划。数十年研究产出复杂的代价优化器，却仍不完美，会选次优计划拖慢执行。主因是不准的基数估计与依赖简单假设的代价模型。现代 DBMS 提供优化器 hint 作为影响/覆盖计划选择的机制。困难在于选对 hint 极复杂，错误 hint 甚至可能反噬性能。近期用 ML 自动化此过程的趋势有两条：完全替换传统优化器，或以 hint/代价修正引导现有优化器。尽管进展鼓舞人心，基于 LLM 的技术因推理成本高、延迟大，离真实部署仍远。本文提出一种直白方法：利用 LLM 对「执行计划」（而非查询本身）的嵌入来识别最优 hint 集，结合 LLM 表示的表达力与一个简单、快速、基于类比的预测系统，无需训练， bridging LLM 的表达力与对快速高效优化系统的需求。

3 LLM-PM

我们把自动 hint 选择形式化为：给定查询 Q，找 hint 集 H 使 T(Q, P_H) 最小（公式 1）。H(Q) 为所有有效 hint 配置空间，实践中极大。为加速，用两种技术：超时阈值（每次执行后 Tmin←min(Tmin,T)，超则中止并标记次优）与计划缓存（相同物理计划复用已测延迟）。考虑的 7 个布尔 hint 为 enable nestloop / hashjoin / mergejoin / seqscan / indexscan / indexonlyscan / bitmapscan（2⁷=128 集）。等价 hint 集消歧：选「禁用算子最少」的规范配置，以保留优化器内置经验。

自适应 hint 搜索（Algorithm 1）单遍扫描全部 128 集并持续收紧超时；先用全启用执行建立 Tmin 与计划缓存 C，其余配置在 w=8 线程并行；每 hint 集产生计划，若已在 C 则取延迟否则在 Tmin 超时下执行，超时标记次优，更快完成则原子更新 Tmin。因慢计划被快停、重复计划经缓存短路，过程把算力集中在少数有希望的候选上，同时保证全覆盖。

3.3 Plan-Mapping 算法

核心假设：相近嵌入空间中的计划共享物理结构，故同一 hint 常在该局部邻域有效。离线搜索产出三元组 (d_i, H*_i, o_i)。两阶段：① 邻域投票——在 N 个最近默认计划中，选最常见的非默认 hint 集作 Hcand；② 一致性校验——用 Hcand 重规划，对默认计划（p0）与候选计划（pcand）各取 K 个最近邻，比较两邻域平均运行时，若候选邻域更小则采纳。超参 N=16、K=16、欧氏距离（Algorithm 2）。

5 结论

我们提出 LLM-PM，一个轻量、免训练、利用预训练 LLM 计划嵌入经 hint 推荐引导传统代价优化器的框架。它先在 N 个最近默认计划中找本地流行 hint 集，再用双邻域一致性校验（比较默认与优化计划区域平均运行时）验证收益，无需改动底层 DBMS 或大量离线训练。在 openGauss 的 JOB-CEB 上证明其一致的性能改进。未来可藉度量学习或计划-SQL 混合特征精炼嵌入空间，并按局部密度动态适配邻域大小，在保持简洁实用的同时进一步缩小与最优 hint 选择的差距。
