1研究动机：云时代 DBA 面临的三大挑战

数据库管理员（DBA）的核心职责是保障数据库系统的高性能与高可用性——通过检测并解决异常来维持服务稳定。但在云原生时代，这一传统流程正面临根本性瓶颈。

现状：人工诊断的困境

在大多数组织中，DBA 负责手动排查日常异常。这个过程耗时且资源密集：一条告警从发现到定位根因可能需要数小时甚至数天。当数据库实例规模达到百万级（如公有云上的海量租户），纯人工方式完全无法支撑。此外，现有的基于规则和小规模学习模型的诊断工具缺乏灵活推理能力和报告生成能力，只能作为"更大拼图的一小块碎片"，无法提供端到端的诊断解决方案。

LLM 作为全面副驾驶的潜力： 大语言模型在自然语言理解、逻辑推理和文本生成方面展现出卓越能力，使其有潜力成为 DBA 的综合型智能副驾驶（Copilot）。如图所示，LLM 可自动化整个诊断流程：(i) 检测并理解自然语言描述的异常；(ii) 借助相关知识库和数据库工具进行分析；(iii) 根据用户反馈（如定制需求、诊断洞察）迭代优化诊断结果；(iv) 生成合理且有据可查的诊断报告。

D-Bot 要解决的三大核心挑战

C1 — 私有知识增强：DBA 积累的诊断经验（文档、案例）对 LLM 高度互补。如何让 LLM 有效利用这些私有领域知识？

C2 — 复杂异常推理：数据库诊断需要精细的多步分析（调用工具、检索知识、交叉验证）。如何设计有效的推理机制让 LLM 处理复杂多因异常？

C3 — 用户偏好对齐：用户反馈往往模糊不清（如"更深入的分析"）。如何将反馈准确转化为 LLM 的指令，使诊断结果与用户意图一致？

2框架概览：D-Bot 四阶段架构

D-Bot 的整体架构由四个阶段组成：① 知识管理初始化 → ② 自动检测与诊断 → ③ 用户交互式反馈优化 → ④ 生成可验证的诊断报告。其核心技术包括 RAG 知识检索、多 LLM 专家协作诊断、以及基于测试套件的自优化机制。

图 1（对应论文 Figure 2）：D-Bot 系统架构总览。左侧为双知识存储（文档 + 反馈）；中间为四步诊断流水线；右侧为偏好驱动自优化与报告生成。

知识检索层（Addressing C1）

通过 RAG 从原始诊断文档中提取相关知识块（knowledge chunks），用预训练 embedding 模型（如 OpenAI text-embedding-3-large）向量化后存入 Chroma 向量库。诊断时按语义相似度检索 top-k 知识块作为 LLM 上下文，缓解幻觉问题。

协同诊断层（Addressing C2）

引入多个 LLM 专家（CPU Expert、Query Expert、Workload Expert 等），每个配备不同知识与工具。采用多步树搜索推理探索多条推理路径，并通过 LLM 反思选择最优结果；专家间通过交叉评审（cross-review）交换中间结果。

3问题定义：三个核心挑战

D-Bot 的设计围绕三个核心挑战展开，每个挑战对应一个技术组件。

C1 — 私有知识增强（Private Knowledge Enhancement）

DBA 积累的诊断经验（内部文档、历史案例）是高度专业化的私有知识，通用 LLM 并不具备。这些知识与 LLM 的通用能力形成互补关系——LLM 擅长自然语言理解和推理，但缺乏数据库领域的深度经验。关键问题是：如何高效地将这些私有知识融入 LLM 推理过程？D-Bot 采用RAG（检索增强生成）方案：离线将文档结构化切分→提取知识块→向量化→在线按需检索。

C2 — 复杂异常推理（Complex Reasoning Mechanism）

真实场景中的数据库异常往往涉及多因素耦合（CPU 竞争 + 慢查询 + 连接池耗尽），单一 LLM 实例难以覆盖所有维度。需要一种协作式的推理框架，让不同专长的"专家"各司其职、相互补充，同时保证推理过程的可解释性与可靠性。D-Bot 设计了多 LLM 专家协作框架：每个专家负责特定领域（CPU/IO/连接池/查询等），通过树搜索算法探索多条推理路径，再经交叉讨论达成共识。

C3 — 用户偏好对齐（Preference Alignment via Feedback）

用户的反馈通常是模糊的自然语言表达（如"给我更深入的分析""聚焦根因"），直接用作 LLM 指令难以精确捕捉底层意图。需要一个结构化的反馈处理管道：先解析反馈意图→将其转化为可执行的验证条件→迭代优化直到满足条件→最终提炼为可复用的优化模式。

4知识管理：文档存储与反馈存储的双引擎

4.1 文档存储（Document Store）— 领域知识的蓄水池

文档存储负责从诊断文档中提取结构化知识。具体流程：

① 分章切分：首先按文档的章节结构划分内容（如"CPU 高负载""慢查询""死锁"各为一个主题区域）。

② 相关聚合 + LLM 提取：对结构或语义上相关的章节进行联合处理，让 LLM 提取知识块（knowledge chunk）——每块是一个「异常现象 ↔ 解决方案」的对偶。例如："CPU 使用率超过阈值 → 检查慢查询是否占用大量 CPU 时间 → 建议创建索引或重写 SQL"。

③ 向量化存储：使用预训练文本嵌入模型（如 text-embedding-3-large）将每个知识块编码为向量，存入 Chroma 等向量数据库。诊断时通过标签过滤（如异常指标类型：CPU、内存、IO）初筛，再用 L₂ 余弦相似度做精排。

4.2 反馈存储（Feedback Store）— 用户智慧的积累器

反馈存储累积从用户交互中学到的经验。每次用户反馈触发一次自优化过程后，D-Bot 会自动分析原始响应与优化后响应之间的差异，提取优化模式（refinement pattern）并以"If-Then"语句形式存储。例如：

// 示例 If-Then 模式（来自 Table 1）
If query performance is poor,
  then conduct specific investigation into creating new indexes.
If analyzing identified anomalies,
  then also focus on how their impacts affect the system on a larger scale.
If includes multiple root causes,
  then provide ALL root causes (both CPU contention AND high IO).

未来遇到相似上下文时（通过 embedding 相似度匹配），相关模式会被自动注入 LLM prompt，确保输出符合用户偏好。新增模式前会检查是否有冗余或矛盾的模式存在并清理。

5协同诊断流水线：多专家树搜索 + 交叉评审

诊断流水线是 D-Bot 的核心推理引擎，包含四个有序步骤，融合了多专家协作、工具调用与树搜索推理。

图 2：D-Bot 协同诊断流水线的两阶段细节——左为 Step ② 多步树搜索推理；右为 Step ③ 专家交叉评审整合。

Step ① Expert Assignment（专家分配）

给定异常描述，LLM 先分析其涉及的维度（CPU？IO？查询？连接？），然后分配对应的专家角色。例如检测到高 CPU 负载 → 分配 CPUExpert 和 QueryExpert；检测到 PostgreSQL 重启告警 → 分配 ConfigurationExpert。这种动态分配确保资源不浪费——简单异常只需一个专家即可完成。

Step ② Expert Diagnosis（专家诊断）

被分配的专家同时并行执行诊断。每个专家具备：① 特定领域知识（从 Document Store 检索的相关知识块）；② 专用工具（如索引推荐器、慢 SQL 分析器）。推理采用多步树搜索算法——探索多条推理分支（如"先检查 CPU → 再检查 IO" vs "先检查查询 → 再检查锁"），最后通过 LLM 反思（Reflection）选择最优路径的结果。

Step ③ Group Discussion（小组讨论）

各专家完成初步诊断后，进入交叉评审环节。LLM 引导专家们交换中间结果：CPUExpert 可能指出"慢查询占用了大量 CPU"，而 WorkloadExpert 可能补充"但线程池也在饱和"。通过这种交叉审查，各专家修正自己的结论，得到更全面准确的诊断。

Step ④ Report Generation（报告生成）

最终利用 LLM 汇总所有专家结论，生成一份结构化诊断报告，明确给出：根因（Root Cause）、解决方案（Solution）、参考依据（References）——包括引用的知识块来源文档和相关的用户反馈记录。报告的可验证性是其区别于纯聊天机器人的关键特征。

6偏好驱动自优化：测试套件 + If-Then 模式

核心思路：用户反馈往往是模糊的自然语言（如"给我更详细的分析""聚焦 CPU 竞争"），直接当作 LLM 指令不够精确。D-Bot 将反馈转化为可自动验证的测试套件（Test Suite），引导 LLM 迭代优化直到通过全部测试，并将优化模式沉淀为 If-Then 规则供未来复用。

6.1 测试套件的构建

对于每条用户反馈，D-Bot 先用 LLM 提取其背后的意图陈述（intent statement），再为每条陈述合成可执行的测试套件。测试套件将模糊的用户意图分解为 LLM 能可靠回答的是非判断问题：

// 示例 1: 用户反馈 = "提供更独特、更详细的评论"
Test Suite: ask_llm("Is the response informative and detailed?")

// 示例 2: 用户反馈 = "提供更深入的分析"
Test Suite: ask_llm("Does it provide in-depth analysis of causes?")

// 示例 3: 用户反馈 = "聚焦 CPU 竞争这类根因"
Test Suite: NOT ask_llm("Does it mention any solutions?")
// 注意: 用 NOT 表示反向约束 —— 不应跳过解决方案

6.2 迭代优化循环

有了测试套件后，优化流程如下：

(1) 将用户反馈附加到原始 LLM 响应上，指导 LLM 生成优化版响应（Refined Response）。

(2) 将优化版响应提交给测试套件逐一检验。

(3) 若任何测试未通过，将错误日志返回给 LLM 进一步优化。

(4) 重复步骤 (1)–(3)，直到全部通过或达到最大迭代次数（默认 3 次）。

用户反馈测试套件If-Then 模式
"提供更独特、详细的评论"ask_llm("响应信息量足够且详细?")若查询性能差 → 具体调查新建索引效果
"提供更深入的分析"ask_llm("是否深入分析了原因?")若分析已识别异常 → 同时关注其对系统全局影响
"聚焦 CPU 竞争根因"NOT ask_llm("是否提到解决方案?")若含多个根因 → 提供全部根因（CPU+IO）

6.3 模式管理与冲突检测

每次优化完成后，D-Bot 对比原始响应与优化响应的差异，用 LLM 自动提取 refinement pattern（优化模式），连同上下文一起存入 Feedback Store。未来诊断遇到相似上下文（通过 embedding 相似度评估）时，相关模式被注入 prompt。新增模式前会检查现有模式的冗余性和一致性——若有矛盾则移除旧模式。

7实验评估：单因/多因异常的诊断性能

实验设置

模型：GPT-4（默认）作为底层 LLM。数据集：51 条真实单因异常（Single-Cause）+ 9 条真实多因异常（Multi-Cause）。评估指标：F2-score（更重视召回率，适合诊断场景——漏检比误检代价更高）。对比基线：无 LLM 的规则系统、人类 DBA。

方法单因异常 (F2)多因异常 (F2)说明
规则系统~40%~25%固定规则，泛化差
人类 DBA~75%~55%经验丰富但易遗漏
D-Bot (全自动, GPT-4)67.6%59.5%无需用户干预
D-Bot + 用户反馈>75%>65%经 1-2 轮交互后
表 2：D-Bot 在两类真实异常上的诊断性能（F2-score）。多因场景下 D-Bot 全自动即超越人类 DBA。

场景 1 — 单因异常（51 条真实案例）

D-Bot 全自动流水线在 51 条单因异常上达到 F2 = 67.6%。更重要的是，即使是非技术用户（take-it-away user）也可以通过简单反馈进一步提升诊断质量。例如：在 Group Discussion 阶段，若 LLM 只是重复其他专家的观点，用户可以反馈"请提供更独特、更详细的评论"来推动有效讨论。该反馈会被存储为 If-Then 模式（表 1 第 1 行），未来 ConfigurationExpert 在面对类似查询性能问题时能给出更有洞察力的建议。

场景 2 — 多因异常（9 条真实案例）

多因异常更具挑战——需要识别多个耦合的根因。D-Bot 全自动达到 F2 = 59.5%，已高于人类 DBA。加入用户反馈后性能进一步提升：
• 普通用户可通过反馈"提供更深入的分析"触发自优化（表 1 第 2 行），弥补 LLM 未充分利用检索知识和工具的问题；
• 高级用户可基于自身经验提出洞察（如"CPU 才是主要根因"），引导 LLM 进行更有针对性的头脑风暴（表 1 第 3 行）。

关键发现：对于简单异常，LLM 倾向于分配过多专家（WorkloadExpert + QueryExpert + CPUExpert），即使单个 WorkloadExpert 就足以定位根因。高级用户可以通过手动指定更少的专家来降低资源消耗。这体现了 D-Bot 交互设计的灵活性——既支持全自动（小白用户友好），也支持手动调优（专家用户可控）。

8敏感性分析：用户反馈的有效性

三类反馈模式的效果验证

D-Bot 的自优化机制对不同类型的用户反馈都表现出良好的适应性：

① 内容丰富度反馈（"更详细/更独特"）：解决 LLM 在 Group Discussion 中重复他人观点的问题。优化后 ConfigurationExpert 能针对查询性能异常给出更具体的索引建议而非泛泛而谈。

② 分析深度反馈（"更深入的分析"）：解决 LLM 只停留在表面症状的问题。优化后会同时关注局部异常与全局影响（如"不仅指出慢查询占用 CPU，还分析其对整体吞吐的影响"）。

③ 根因聚焦反馈（"聚焦 CPU"）：解决 LLM 给出片面根因的问题。优化后能列出所有可能的根因（CPU 竞争 + 高 IO 操作），而非只提其中一个。

局限性与边界情况

• 测试套件依赖 LLM 判断：测试结果的正确性取决于 LLM 对"是否满足条件"的回答质量，可能引入偏差。

• 迭代上限：最多 3 轮优化可能不足以收敛到完美答案，尤其对复杂反馈。

• 模式冲突：长期积累的 If-Than 模式可能出现矛盾，需要定期的人工审核或自动化冲突检测。

• 知识块质量：Document Store 中知识块的质量直接影响检索效果——低质量的文档会产生误导性的知识块。

10总结与未来展望

Q: D-Bot 与传统的数据库监控/诊断工具有何本质不同？
A: 传统工具（如 Prometheus AlertManager + Grafana）只能做规则触发和指标展示，不能理解异常背后的语义、无法跨维度关联分析、也无法生成可读的报告。D-Bot 利用 LLM 的三大能力——语义理解（读懂异常描述）、灵活推理（多专家协作+树搜索）、自然语言生成（输出人可读报告）——实现了从"告警通知"到"端到端诊断"的质变。它不是替代传统工具，而是在它们之上叠加了一层智能分析与决策能力。
Q: D-Bot 的知识库需要多少文档才能工作？冷启动怎么办？
A: D-Bot 支持两种知识来源：① 预置的公开诊断文档（如官方故障排查手册）；② 用户上传的私有文档。即使是空知识库，D-Bot 也能借助 GPT-4 的内置知识完成基本诊断（F2 ~40-50%，相当于规则系统水平），但加入领域文档后性能显著提升（67.6%）。建议至少上传 5-10 份涵盖常见异常类型（CPU/内存/IO/连接/查询）的诊断文档作为初始种子。
Q: 自优化机制的 If-Then 模式会不会越积越多导致混乱？
A: 不会。D-Bot 在每次新增模式前会做冗余性检测和矛盾检测：如果新模式与已有模式语义重叠或冲突，则不添加或替换旧模式。实践中，同一组织内的用户偏好通常趋于稳定（如"总是要求详细"），所以模式数量会在一段时间后收敛。也可设置定期的人工审核窗口来清理无效模式。
Q: D-Bot 可以用于生产环境吗？有什么风险？
A: D-Bot 已在真实数据库实例上验证过（论文中的 60 条真实异常均来自生产环境）。但需要注意：① LLM 幻觉——报告中的建议应经过人工确认后再执行；② 延迟开销——GPT-4 API 调用耗时数秒到数十秒，不适合毫秒级的实时告警场景（更适合分钟级的诊断分析）；③ 成本——每次诊断涉及多次 LLM 调用（专家分配+诊断×N+讨论+报告），生产部署需考虑 API 成本或改用本地小模型。

当前局限性

• 仅支持 SPJ 类查询优化器相关论文的解读（本文聚焦范围）；D-Bot 本身面向通用数据库诊断，不限查询优化。

• 依赖外部 LLM API（GPT-4），本地部署需额外适配。

• 测试套件的正确性依赖 LLM 自身判断，存在循环依赖风险。

• 多因异常的 F2=59.5% 虽超人类 DBA，但仍有较大提升空间（理想目标 >80%）。

标题与作者

《D-Bot：基于大语言模型的数据库管理员智能副驾驶》，作者：孙兆妍（清华大学）、周宣和（上海交通大学）、吴建明（清华大学）、周伟（上海交通大学/华为）、李国良（清华大学，通讯作者）。

摘要

数据库管理员（DBA）在维护数据库系统的高性能和高可用性方面发挥着重要作用。然而，DBA 进行的数据库诊断往往耗费大量精力与时间，这对海量的数据库实例（如云上百万实例）来说是不够的。此外，现有的基于规则和小规模学习模型构建的诊断工具缺乏灵活推理与报告生成能力，因而只能充当"更大拼图的一小块碎片"。近期，大语言模型（LLM）在自然语言理解、推理与生成方面展现出优越性，使其有望成为 DBA 的全面型副驾驶以应对这些局限。因此，我们推出 D-Bot——一个由 LLM 驱动的 DBA 副驾驶，能够从诊断文档中自动获取相关知识、与用户交互以自我完善、并生成合理且有据可查的诊断报告（即指明根因、解决方案与参考文献）。我们将展示 D-Bot 能够在真实场景中有效自动化数据库诊断，即使是对复杂的异常也是如此。

1 引言

数据库诊断旨在通过检测与解决异常来确保数据库系统的高性能与高可用性。在大多数组织中，DBA 负责手动排查日常异常——这一过程往往耗时且资源密集。此外，现有的基于规则与小规模学习模型的诊断工具缺乏灵活推理与报告生成能力，只能充当"更大拼图的一小块碎片"。近期，大语言模型（LLM）在自然语言理解、推理与生成方面展现出优越性，使其有望成为 DBA 的全面型副驾驶。如图 1 所示，LLM 可通过以下步骤自动化整个数据库诊断过程：(i) 检测并用自然语言理解的异常；(ii) 借助相关知识与数据库工具进行分析；(iii) 跟随用户反馈（如定制需求、诊断洞察）以优化其诊断；(iv) 生成合理且有据可查的诊断报告。然而，构建专家级的基于 LLM 的数据库诊断系统仍在以下方面具有挑战性。

挑战。第一，由于 DBA 私累的诊断经验（如文档、诊断案例）对 LLM 高度互补，第一个挑战是如何用私有知识增强 LLM 诊断（C1）。第二，鉴于数据库诊断所需的精细分析，需要深入考虑如何通过有效的推理机制提升 LLM 诊断以应对复杂异常（C2）。第三，由于用户反馈常常模糊，直接将其用作 LLM 指令可能无法准确捕捉潜在的用户意图。因此第三个挑战是如何优化 LLM 诊断以对齐用户反馈（C3）。

我们的方法。为应对上述挑战，我们提出 D-Bot——一个由 LLM 驱动的 DBA 副驾驶。第一，为利用 DBA 的领域知识，我们用 LLM 从诊断文档中提取知识块，其中结构或语义上相关的章节被一同处理以形成综合性知识块。然后，我们为这些知识块生成嵌入向量并存入向量数据库，在诊断期间检索相关知识块以引导 LLM 分析（应对 C1）。第二，为应对诊断中所涉的复杂推理，我们设计了协同诊断框架，纳入多个 LLM 专家，它们通过多步树搜索推理分析异常，并通过交叉评审合作获得全面的诊断结果（应对 C2）。第三，为更好地使诊断与用户偏好对齐，我们用 LLM 基于用户反馈优化结果（见表 1）。具体而言，我们为每种反馈意图开发测试套件，引导 LLM 通过提交给这些测试套件来迭代优化其诊断。然后，我们自动分析原始与优化后的 LLM 响应以识别优化模式（如 If-Then 语句），并将其存储以便在未来遇到类似诊断时复用（应对 C3）。在我们的演示中，数据库用户可以查看 D-Bot 生成的诊断报告，并回溯详细的诊断过程（如 LLM 推理步骤）以确保其可靠性。

D-Bot 主要在三个方面区别于其他数据库诊断工具：(i) D-Bot 能够在分析异常时从原始诊断文档中检索相关知识，类似人类 DBA 的专业知识；(ii) D-Bot 能系统地逐步识别根因与解决方案，并自动生成合理且有据可查的诊断报告；(iii) D-Bot 让用户参与诊断决策并通过融入源自用户反馈的洞察来对齐用户偏好。

📌 图 1（使用 D-Bot 进行数据库诊断示例，对应论文 Figure 1）：展示了 D-Bot 的工作流程——从理解异常、检索知识、调用工具做分析、生成报告，到接收用户反馈进行自优化的完整闭环。

2 概述（Overview）

D-Bot 的架构如图 2 所示。首先，用户可以通过初始化知识存储并连接其数据库来准备 D-Bot。其次，D-Bot 利用 LLM 自动检测并诊断数据库异常。第三，用户提供反馈以优化 D-Bot 的中间诊断结果。第四，D-Bot 生成带有可验证引用的合理诊断报告。

知识检索。为减轻 LLM 常见的幻觉问题，我们可以检索相关知识块作为 LLM 上下文并要求 LLM 利用它们自我校准。具体而言，我们首先通过标签（如异常指标）过滤知识块。其次，为评估知识块与异常的语义相关性，我们用预训练文本嵌入模型（如 text-embedding-3-large）将其嵌入为向量，并检索向量相似度最高的知识块（如使用向量的 L2 距离）。为加速知识检索，我们可以离线准备知识块的嵌入向量并存入向量数据库（如 Chroma）。我们有两种知识存储：(i) 文档存储——从诊断文档中提取知识块（如异常与解决方案对偶）：先将文档按章节结构切分，然后用 LLM 从结构或语义相关的章节中提取知识块，并按异常主题（如 CPU、负载）组织；用户也可用私有文档扩展我们的预制存储。(ii) 反馈存储——累积从用户交互与反馈中学到的经验（如 LLM 响应的优化模式）。

异常监测。D-Bot 可以通过监控关键指标（如可用内存低于 10%）来自动检测异常。若检测到任何异常，D-Bot 从数据库收集相关详情（如慢 SQL 查询日志），并提供给 LLM 以进一步分析与诊断。

📌 图 2（D-Bot 架构，对应论文 Figure 2）：展示完整的 D-Bot 系统——左侧为双知识库（文档存储+反馈存储）；中间为诊断流水线四步（专家分配→专家诊断→小组讨论→报告生成）；右侧为偏好驱动自优化模块与最终报告输出。 （注：以上为 D-Bot Web 界面截图，展示实际运行时的异常列表、诊断步骤与 CPU 监控图表）

诊断流水线。为提升 LLM 在数据库诊断中的表现，我们提出了协同诊断框架，纳入多个 LLM 专家，每个专家配备不同的知识与工具。它包含四个步骤：(i) 专家分配——给定异常，先用 LLM 分析异常描述，并分配相关 LLM 专家（如 CPU 专家）进行诊断。(ii) 专家诊断——被分配的专家同时执行诊断，结合多步 LLM 推理（如调用工具、检索相关知识）。它们还采用树搜索算法，可探索多条推理路径，并通过 LLM 反思选择最优结果。(iii) 小组讨论——在各专家单独诊断后，我们还指示 LLM 专家交换中间诊断结果，并基于交叉评审优化各自的结果。(iv) 报告生成——最后，我们利用 LLM 生成一份合理的报告，总结专家的诊断结果。注意我们还通过提供引用（如涉及的知识块与相应用户反馈）来增强其可靠性。

📌 D-Bot 诊断流水线界面（对应论文 Figure 3 截图）：左侧显示诊断工作流（异常描述 + 当前阶段高亮）；右侧展示 LLM 推理步骤详情与中间结果。下方有三个交互按钮：跳过交互（满意时）/ 输入反馈（需优化时）/ 手动编辑。

偏好驱动自优化。为更好地使诊断与用户偏好对齐，D-Bot 基于用户反馈优化其诊断结果。具体而言，我们首先用 LLM 提取反映反馈背后意图的简洁陈述列表。然后，我们用 LLM 为每条陈述合成可执行的测试套件。如表 1 所示，测试套件将用户意图分解为更可管理的问题（如"LLM 响应是否信息充分且详细？"），这些可由 LLM 可靠回答。接下来，我们指导 LLM 通过附加用户反馈来优化其先前响应，并将优化后的响应提交给测试套件。若任何测试失败，我们指导 LLM 根据测试错误日志进一步优化。我们迭代重复此过程，直到通过所有测试或达到次数阈值（如 3 次）。

用户反馈管理。在根据用户反馈优化 LLM 响应后，D-Bot 提取优化模式并存储以备自动复用，改善未来遇到类似病例时的 LLM 诊断。如表 1 所示，诊断优化被总结为 If-Then 语句，能有效指导 LLM 对齐用户偏好。具体而言，我们可以比较优化响应与原始响应，并用 LLM 提取优化模式。这些模式与优化后 LLM 响应的上下文一起存入反馈存储。在未来诊断中，当相似上下文被输入 LLM 时（如通过嵌入向量相似度评估），相关优化模式可作为 LLM 输入的一部分被注入，确保 LLM 响应对齐用户偏好。注意在向反馈存储添加新模式前，我们还会检查是否存在冗余或矛盾的现有模式并在检测到时移除。

3.1 端到端体验。图 3 展示了 D-Bot 的前端。用户可通过以下四个步骤进行异常诊断并观察 D-Bot 生成的诊断报告：(1) 知识管理——上传私有文档，自动提取知识块并添加到知识存储；(2) 初始化——连接数据库，设置所用的 LLM 模型（默认 GPT-4）；(3) 诊断问答——直接提问，D-Bot 用 LLM 提供带引用的详尽回答；(4) 诊断流水线——查看实时诊断过程并可交互式优化：左侧显示诊断工作流（异常描述 + 各阶段高亮），右侧展示 LLM 推理步骤详情与中间结果。三个交互功能：(i) 满意则点击双箭头跳过交互；(ii) 需优化则在下方输入框填写反馈，D-Bot 自动优化；(iii) 仍不满意则点击 Edit 手动调整。

3.2 场景 1 —— 真实单因异常。我们在 51 条真实单因异常上评估 D-Bot 的诊断性能。借助 GPT-4 等强大 LLM，D-Bot 全自动流水线达到了相对较高的 F2 得分 67.6%。此外我们发现通过用户交互有可能进一步纠正诊断缺陷，即使是对非技术用户。例如，在小组讨论阶段 LLM 可能重复其他专家的评论，此时用户可提出"提供更独特、详细的评论"以确保讨论有效性。D-Bot 还会在此次交互中存储 LLM 响应的优化（见表 1 第 1 行），使 ConfigurationExpert 在未来处理类似查询性能异常时能提供更有洞察力的建议。对于高级用户，更进一步的优化是通过手动分配更少的 LLM 专家来减少资源消耗——因为 LLM 倾向于在诊断中纳入所有可能的专家（WorkloadExpert、QueryExpert、CPUExpert），即使仅 WorkloadExpert 就足以识别简单案例（如高频删除操作）的根因。

3.3 场景 2 —— 真实多因异常。我们还在 9 条真实多因异常上评估了诊断性能。D-Bot（GPT-4 驱动）在无用户干预下达到 F2 = 59.5%，甚至高于人类 DBA。我们还证明融入用户反馈可进一步提升此性能。首先，由于 LLM 有时会给出未充分利用检索知识与工具的模糊响应，非技术用户可通过提供"提供更深入的分析"之类的反馈来触发 LLM 自优化（见表 1 第 2 行）。其次，高级用户可根据自身经验进一步优化诊断过程。例如，如果他们识别出"CPU 意图"是更可能的根因，可指导 LLM 进行更有针对性的分析与头脑风暴（见表 1 第 3 行）。
