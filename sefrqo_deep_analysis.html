1研究动机：学习型优化器的三重理想属性

SEFRQO 的核心论点是：现有的学习型查询优化器（LQO）最多只能同时满足「从错误中学习」「零冷启动」「自适应变化」这三条理想属性中的两条，而 SEFRQO 是第一个把三者全部集于一身的方案。

三条理想属性（Ideal Properties）

① 从错误中学习（Learn from mistakes）：能快速、主动地把查询计划的执行反馈吸收进来。代表性系统是 Bao——它用上下文多臂老虎机为每个查询挑 hint 集，并通过执行反馈持续重训。

② 立即工作、无冷启动（No cold start）：不需要海量训练数据就能开始产生收益。Fastgres 属于这一类：它用动态规划 + 监督模型为每个查询选 hint，无需长时训练，但它无法在 workload 变化后不重训地自适应。

③ 自适应 schema / workload / 数据（Adapt）：当表结构、查询负载或底层数据变化时，不需大量人工干预即可适应。传统学习型方法（如 Balsa、Neo）一旦数据/schema 变了就得完全重训。

为什么三者难以兼得？ 不使用 LLM 的 LQO（Bao、Fastgres、Lero…）推理能力有限，要么需要大量训练数据（→ 冷启动），要么假设 schema/workload 固定（→ 无法自适应）。而使用 LLM 的 LQO（LLM-QO、LLMOpt、LLMSteer）借助 LLM 能「立即工作、能自适应」，但微调成本太高，无法在 DBMS 关键路径上「即时从错误中学习」——它们把反馈学习推迟到了昂贵的离线重训阶段。

SEFRQO 的破局点

作者观察到：LLM 的上下文学习（in-context learning）能力可以在不修改模型参数的情况下，仅靠在 prompt 里塞入「相似查询 + 本查询的历史执行记录」就让模型即时进化。于是 SEFRQO 把问题拆成两个阶段：离线用微调（SFT + QGRPO）打底（解决冷启动与格式正确），在线用 RAG 提示做自进化（解决从错误中学习与自适应）。一个持续更新的向量数据库充当优化器的「记忆」。

2框架概览：三阶段两范式

SEFRQO 的整体流程（论文 Figure 1）由三个阶段组成，分别对应三个优化子问题：离线 SFT（→ 公式 5，保证语法正确）、离线 QGRPO（→ 公式 6，偏好低延迟）、在线 RAG 提示优化（→ 公式 7，最小化该查询延迟）。

图 1（对应论文 Figure 1）：SEFRQO 系统概览。离线两阶段产出固定参数的微调 LLM；在线阶段通过动态 prompt 让 LLM 持续从执行反馈中自进化。

定义 1：离线微调（公式 5 + 6）

给定 SQL 查询 Q 与相关统计 stats（如表基数），微调 LLM 参数 θLLM，目标有二：① 语法正确（hint 能被优化器接受为合法计划）；② 延迟偏好（生成的 hint 使执行延迟最小）。用 SFT 解决①、用 QGRPO 解决②。

定义 2：在线上下文学习（公式 7）

对同一个固定参数的 LLM，在运行时构造一个「优化后的 prompt」P*，引导它生成使该查询延迟最小的 hint。P* 由本查询的 stats + 从 VecDB 动态检索的 k 条相似参考查询构成。这被形式化为一个 prompt 优化问题。

3预备知识与问题定义

在理解两个阶段前，论文先交代了两个基石：① 基于 hint 的查询计划生成（如何让 LLM 控制 PostgreSQL）；② LLM 的自回归生成（论文用到的概率记号）。

3.1 基于 Hint 的计划生成（pg_hint_plan）

SEFRQO 不重新发明优化器，而是用 pg_hint_plan 在 SQL 注释里写「hint」来控制 PostgreSQL 的计划。

Join-order 模式（简单模式）：用 Leading 子句指定连接顺序。例如 Leading(a (b c)) 表示先做 b⊲⊳c，再做 a⊲⊳(b⊲⊳c)，由 PostgreSQL 自己决定每个连接的算子类型。

Full-plan 模式（完整模式）：进一步指定扫描类型（IndexScan、SeqScan）与连接类型（HashJoin、NestedLoop），精确控制计划。

3.2 LLM 自回归生成

prompt 分为 System Prompt（定义角色，如「你是做查询优化的数据库专家」）和 User Prompt（具体任务）。LLM 把文本 token 化后自回归地逐 token 预测，输出序列 y=[y₁…y_T] 经分词器还原为自然语言——在本文中 y 就是一条 hint。其生成概率可分解为：

pθ(y | x) = ∏t=1T pθ(yt | x, y1:t−1)

论文为简洁起见，用 x 直接表示 prompt、h 直接表示生成的 hint，把 token 化细节隐藏。

4离线阶段一：SFT 监督微调学会 Hint 格式

SFT 的目标是让 LLM「认识任务」——学会输出格式正确、能被数据库接受的 hint。它用一个由黄金 hint 对构成的训练集 Dsft 来最大化条件概率 pθ(h* | x)。

Algorithm 1：SFT 数据集准备（含运行示例）

对每条训练查询 Qi：

① Ei ← DB(Qi)：用 EXPLAIN ANALYZE 拿到执行计划与中间统计（行 2）；

② Ti ← ExtractTables(Ei)：抽取涉及的表（行 3）；

③ Ci ← GetCardinalities(Ti, Ei)：拿到各表基数（行 4）；

④ xQi ← ConstructPrompt(Qi, Ci)：把查询与基数拼成 prompt（行 5）；

⑤ h*i ← ExtractHint(Ei)：从计划里抽出 hint 作为标签（行 6）；

⑥ 把 (xQi, h*i) 加入 Dsft。

具体例子（论文 Figure 2）： 对一条涉及 company_name, company_type, movie_companies, role_type, title 的查询，其计划树经抽取得到 5 张表的根–叶结构。最终抽出的 full-plan hint 形如：
抽取得到的 Hint/*+ NestedLoop(mc cn t rt ct) NestedLoop(mc cn t rt) HashJoin(mc cn t) NestedLoop(mc cn) SeqScan(mc) IndexScan(cn) IndexScan(t) SeqScan(rt) IndexOnlyScan(ct) Leading(((((mc cn) t) rt) ct)) */
其中 Leading(((((mc cn) t) rt) ct)) 编码了连接顺序，其余子句指定了每张表的扫描方式与每个连接的连接算法。SFT 就是让 LLM「照着这种格式学」。

SFT 的训练目标是最小化负对数似然：Lsft(θ) = E(x,h*)∼Dsft [ − log pθ(h* | x) ]

5离线阶段二：QGRPO 延迟奖励强化微调

SFT 只教会「格式」，但没教会「快」。QGRPO（Query Group Relative Policy Optimization）在 SFT 基础上，用「真实执行延迟」作为奖励，把 LLM 的输出偏好推向更低延迟的 hint。它基于 GRPO，但关键创新在于不需要额外的偏好标注数据集。

图 2（对应论文 Figure 3）：QGRPO 工作流——同一查询生成 G 个 hint，逐个真实执行得到延迟，用「相对默认计划的加速比」给奖励，再做组内相对比较更新策略。

5.1 基于延迟的奖励模型（Algorithm 2）

① 先执行不带 hint 的查询得到基线延迟 td；

② 对每个生成的 hint hi，执行得到真实延迟 ti；

③ 计算加速比 Δti = td / ti（越大越好）；

④ 奖励 ri = tanh(ln(Δti))，把奖励压到 [−1, 1]。

为什么用 ln + tanh？

ln 让「加速 2 倍」(Δt=2→ln≈0.69) 与「变慢 2 倍」(Δt=0.5→ln≈−0.69) 对称；tanh 再限制范围，避免极端好/坏计划导致的数值爆炸，提升微调稳定性。

5.2 组内相对优势（公式 9）与简化梯度（公式 10）

优势 Ai = (ri − mean(r)) / std(r)：在「同一查询的 G 个 hint」这个小组内做相对比较，不需要任何人工偏好标注。相比 DPO（需要构造成对偏好数据）、PPO（需要额外的价值模型），QGRPO 只靠真实执行反馈，鼓励 LLM 自我探索，从而在面对未见过 workload 或 schema 漂移时保有更好的推理与泛化能力。

ĜGRPO = Âi + β·( πref(hi|x) / πθ(hi|x) − 1 )

β 控制与参考策略 πref（即 SFT 模型）的 KL 散度，防止更新后策略偏离太远。

6在线阶段：RAG 自进化提示优化

这是 SEFRQO 区别于所有前人工作的核心：用 RAG 让 LLM 在不更新参数的前提下，通过对 prompt 的自我迭代不断降低延迟。论文 Figure 4 把流程拆成三个阶段。

图 3（对应论文 Figure 4）：SEFRQO 的 RAG 自进化三阶段。虚线表示跨轮迭代反馈闭环——每当生成更优计划就更新记忆，下一轮把「历史最佳 + 性能增益」塞回 prompt。

6.1 向量库存储设计（Figure 5）

VecDB 中每条执行记录维护 7 个字段：Id（主键）、Iteration（Preparation/Initialization 为 0，Generation 从 1 起）、Vector（SQL 文本经 SentenceTransformer 编码的嵌入，用于相似度检索）、SQL_id（如 job_33b，用于区分自身与参考）、SQL、Plan、Execution Time。检索时用余弦相似度取 top-k。

6.2 动态 Prompt 结构（Figure 6）

prompt x = (SP ∥ UP)。SP 是静态系统提示（定义角色、可执行的动作、动作约束）；UP 是动态用户提示，包含：① k 条最相似查询及其执行计划作为参考；② 本查询统计 stats（表基数、过滤选择性）；③ 本查询的「历史最佳计划 + 相对默认计划的性能增益 η」；④ 对 LLM 的期望与约束（生成更好的、不要照抄执行引擎或自己历史 hint、格式须合法）。

具体例子（JOB 20a，论文 §6.4）： Preparation 阶段从 PostgreSQL 拿到默认计划 NestedLoop(k mk) → NestedLoop(k mk cc)；Generation 第一轮 SEFRQO 生成 NestedLoop(mk k) → NestedLoop(mk k t)；后续反馈迭代中进化为 HashJoin(mk k) → NestedLoop(mk k cc)。SEFRQO 自己「悟出」两点：① 先 join mk 和 k 有益，且用 HashJoin 比 NestedLoop 更高效；② NestedLoop(mk k t) 这个嵌套循环连接高效，应保留。这正是自进化反馈把「历史最佳计划 + 增益」塞回 prompt 后涌现的行为。

Algorithm 4：跨轮 Prompt 优化

对上一轮 prompt 生成的 hint h 执行得 (plan, t)；取 VecDB 中历史最佳 (plan*, t*)；若 t < t* 则更新历史最佳；取 Initialization 阶段存的基线 t₀，算增益 η = (t₀ − t*) / t₀；把 (plan*, η) 合并进 UP；拼回 SP 得到本轮优化 prompt P*。η<0 表示当前最佳计划比基线还差。

7实验评估：静态 / 动态 / 跨域场景

实验在 AMD Ryzen 9700X + 128GB + A6000 上进行，PostgreSQL 16.4 + pg_hint_plan，向量库用 Milvus，嵌入用 SentenceTransformer(Paraphrase-MiniLM-L6-v2)。评测 6 个本地 LLM：LLaMA3.1-3B/8B 及其 SFT 版(LS3B/LS8B)、SFT+QGRPO 版(LSG3B/LSG8B)，k=1。

7.1 三个研究问题与对比基线

基线：Bao（steering 类 LQO）、Balsa（generative 类 LQO）、LLMSteer（唯一可复现的 LLM-based LQO）。LLM-QO 与 LLMOpt 因无完整代码未能复现。

三种变体：默认（full-plan hint + 向量库参考）、JO（只用 Leading 连接顺序）、NR（只用本查询历史记录、不检索参考查询）。

指标：相对 PostgreSQL 的性能增益（Overall / Filtered）、相对执行时间 RET（↓）、同质性比率 HR（↓，越低表示生成的 hint 越多样、越少照抄默认计划）。

8敏感性分析

论文通过消融与额外实验，量化了各设计选择的影响，并验证了鲁棒性。

生成模式：full-plan vs JO

LSG3B-JO 的同质性比率高达约 70%（几乎只复读默认计划），CEB Overall 增益仅 24.81%，远低于 full-plan 的 50.13%。说明仅给连接顺序约束过于受限，full-plan 模式显著更有效。

参考查询数量 k

k=1/2/3 的 CEB Overall 增益分别为 53.93% / 57.75% / 57.99%，HR 为 0.26 / 0.20 / 0.14。k 增大让 hint 更多样，但增益非单调——太少缺关键信息、太多反而迷惑 LLM。k=2~3 较优。

参考查询的作用（NR 变体）

LSG3B-NR（不检索参考）CEB Overall 仅 41.26%，低于带 RAG 的 50.13%，证明「相似查询作为 in-context 样例」确实有效。

相似度计算方法

默认余弦相似度比内积 / L2 平均高约 3% 增益；内积产生的计划多样性最高（50 轮多生成 150 个不同计划）。

鲁棒性

基数不准：随机扰动过滤/表基数后，LSG8B 仍达 51.50% Overall 增益（仅轻微下降），与静态实验相当。数据漂移：在原始 IMDB 与「仅 2000 年后」缩减库间每 5 轮切换，会出现尖峰（小库反馈干扰自进化），但 LS8B 偶尔反而取得最低延迟。开销：LS3B 推理约 236ms(A6000)/160ms(A100)，RAG 检索平均 0.7ms；CEB 平均省时 520.67ms vs PostgreSQL 922.86ms，端到端仍更低——因此 SEFRQO 面向长时 OLAP 查询。

10总结与未来展望

Q：SEFRQO 最重要的贡献是什么？
它是首个把 RAG 用于查询优化的工作，并用「离线微调 + 在线自进化」两阶段范式，让 LLM 型优化器第一次同时满足「从错误学习 / 零冷启动 / 自适应变化」三大理想属性。
Q：QGRPO 相比 DPO / PPO 好在哪？
PPO 需额外价值模型（内存与算力开销大）；DPO 需人工构造成对偏好数据（标注成本高）。QGRPO 基于 GRPO，对同一查询生成多个 hint、用真实执行延迟做组内相对比较，既无价值模型、也无标注数据，还能借助真实执行环境提升推理能力。
Q：局限与未来？
当前面向 OLAP 长查询——推理 + RAG 开销对 OLTP 短查询仍偏高，留作未来工作。此外 QGRPO 的 RL 特性带来不确定性，8B 模型有时需多次微调才稳定（如 LSG8B 在某些设定下未优于 LS8B）。未来可探索与系统级 knob tuning 的结合（论文指出二者常被视作正交、少有融合）。

一句话总结

SEFRQO 把「查询优化」重新定义为 LLM 的两阶段问题——离线用 SFT+QGRPO 把模型调成「会写且写得快」的 hint 生成器，在线用 RAG 把执行反馈变成不断优化 prompt 的「记忆」，使 LLM 既能冷启动、又能随 workload 与 schema 持续进化。

摘要（Abstract）

查询优化是数据库系统中一个已被研究数十年的关键问题。学习型查询优化器（LQO）能随时间通过吸收反馈来提升性能；然而它们存在冷启动问题，并且往往需要在 workload 漂移或 schema 变化时重新训练。近期的基于 LLM 的查询优化器利用预训练与微调的 LLM 来缓解这些挑战，但它们忽视了 LLM 的上下文学习能力以及「把执行记录作为反馈用于持续进化」的潜力。本文提出 SEFRQO——一个自进化的、基于微调 RAG 的查询优化器。SEFRQO 通过检索增强生成（RAG）框架持续从执行反馈中学习，从而缓解 LQO 的冷启动问题。我们同时采用监督微调（SFT）与强化微调（RFT），使 LLM 能够生成语法正确且性能高效的查询 hint。此外，SEFRQO 利用 LLM 的上下文学习能力，通过动态构造「引用相似查询 + 同一查询历史执行记录」的 prompt 来持续进化，这一自进化范式迭代地优化 prompt，以最小化查询执行延迟。实验表明，SEFRQO 优于当前最优的 LQO，在 CEB 与 Stack 负载上相比 PostgreSQL 分别取得最高 65.05% 与 93.57% 的查询延迟降低。

📌 对应图 1（Figure 1: System overview of SEFRQO）：三阶段流程——① SFT（从标注数据学 hint 格式）；② QGRPO（用奖励函数把 LLM 输出偏好导向低延迟）；③ 在线 RAG 提示优化（用动态 prompt 生成最终 hint）。离线阶段产出 Fine-tuned LLM，在线阶段经 VecDB 与执行引擎形成自进化闭环。

1. 引言（Introduction）

查询优化是数据库系统中基础且性能关键的问题，关注于把声明式用户查询翻译成高效的查询计划。人类专家花费数十年才设计出并改进经典查询优化器（如 PostgreSQL）。为了加速优化器开发并提升查询性能，若干研究把机器学习、大模型等技术应用于查询优化。

尽管这些学习型查询优化器（LQO）令人振奋、甚至已有工业落地，我们认为现有的 LQO 最多只能捕捉三条理想属性中的两条：① 应从错误中学习，快速、主动地吸收查询计划的反馈；② 应当立即可用，无需大量训练（即「冷启动」问题）；③ 应能在 schema、workload、数据变化时自适应，而无需大量人工。

例如，Bao 能从错误中学习并适应变化，但需要数小时训练数据才能具备竞争力；Fastgres 没有冷启动问题且能从错误学习，却无法在 workload 显著变化时无需重训地适应。三个近期应用 LLM 的系统（LLM-QO、LLMOpt、LLMSteer）用微调 LLM 作为计划生成或选择的预言机，但由于微调成本高，它们无法在运行时立即从错误中学习或持续改进。

为什么没有系统同时具备所有理想属性？直觉上：不用 LLM 的 LQO 推理能力较弱，要么需要大量训练数据（→ 冷启动），要么假设 schema/workload 固定（→ 无法自适应）；用 LLM 的 LQO 能立即可用、能自适应，但用现有技术纠正其错误对 DBMS 关键路径而言过于昂贵。

本文提出 SEFRQO——一个具备全部三条属性的、基于 RAG 的查询优化器。SEFRQO 用 LLM 避免冷启动、适应变化；为从错误学习，它用一个持续更新的、存放过往查询执行的向量数据库作为优化器的「记忆」。SEFRQO 生成基于 hint 的查询计划来引导经典数据库系统（如 PostgreSQL）里的优化器，hint 由 LLM 优化并生成。SEFRQO 依赖两个互补阶段来优化 hint 生成：离线 LLM 微调与在线 LLM 上下文学习。由于这些技术比传统优化器开销更大，SEFRQO 面向长时 OLAP 查询，其性能收益可抵消增加的优化开销。

为使微调后的 LLM 能生成格式正确且高效的 hint，SEFRQO 采用「监督微调（SFT）+ 强化微调（RFT）」组合。SFT 教 LLM 期望的 hint 格式；RFT 提出一种高效的 Query Group Relative Policy Optimization（QGRPO），基于 GRPO，把 LLM 输出偏好导向能降低延迟的 hint。QGRPO 定义了一个基于延迟的奖励模型，与数据库执行引擎交互以直接测量所生成 hint 的对应延迟；它还在同一查询生成的多个 hint 之间做组内比较，避免了像 DPO 那样需要额外标注数据集，从而鼓励 LLM 自我探索、提升应对未见过 workload 或 schema 漂移的推理与泛化能力。

另一关键创新是在线 LLM 基于 RAG 的上下文学习：LLM 通过以少量相关 RAG 样例为条件（直接放进输入上下文即 prompt 中）来适应新任务，而无需修改模型参数。关键是检索到的样例要与目标查询高度对齐。为此 SEFRQO 建立并维护一个存放「最相关查询执行记录」的专用向量数据库，并提出自进化反馈范式，让 LLM 从同一查询的历史执行记录中学习——历史最佳计划及其相对基线的性能增益提供了额外上下文，这些组件迭代地优化 prompt，促使 LLM 生成最小化执行延迟的优化 hint。

实验表明 SEFRQO 持续优于 PostgreSQL 与两个 SOTA LQO（Bao、Balsa）。在 CEB 上相比 PostgreSQL 降低 65.05% 执行时间；Stack 上某些查询最高降低 93.57%、整体 40.10%；TPC-DS 上整体降低 13.28%。凭借动态 prompt 与自进化范式，SEFRQO 在未见过的 Stack 负载上展现出惊人的跨域能力，某些指标甚至超过在该负载上直接训练的 LQO，且支持灵活 workload 切换与跨 DBMS（MySQL 上无需额外微调即降低 26.28%）。我们还观察到 SEFRQO 能在子计划层面从给定计划学习并推理出改进计划，并做了开销分析与消融实验。贡献总结为：① 首次把「基于 LLM 的 hint 优化」形式化为离线微调 + 在线上下文学习两阶段；② 提出高效 SFT 流程教 LLM hint 格式；③ 提出 QGRPO（基于延迟的奖励模型 + 组内相对优势）；④ 提出基于 RAG 的 prompt 优化（首次把 RAG 用于查询优化）；⑤ 实验证明 SEFRQO 跨多 workload / 多 DBMS 泛化且无需重训。

2. 预备知识（Preliminaries）

2.1 基于 Hint 的计划生成。 许多 LQO（如集成 PostgreSQL 的 Balsa）用 pg_hint_plan 通过在 SQL 注释里写 hint 来调整执行计划。hint 支持连接顺序模式（简单，用 Leading 指定连接顺序，如 Leading(a (b c)) 表示先 b⊲⊳c 再 a⊲⊳(b⊲⊳c)）与完整计划模式（额外指定扫描类型 IndexScan/SeqScan、连接类型 HashJoin/NestedLoop）。SEFRQO 两种模式都用。

2.2 LLM 生成。 prompt 含 System Prompt（定义角色/上下文）与 User Prompt（具体任务）。文本经 tokenizer 转成 token 序列 x=[x₁…xₙ]，LLM 以自回归方式逐 token 预测，输出 y=[y₁…y_T] 再还原为自然语言——在本文中 y 即 hint。生成概率 pθ(y|x) = ∏t pθ(yt|x, y1:t−1)。

2.3 监督微调（SFT）。 用高质量标注数据 (x, h*) 调整 LLM 参数，最大化 pθ(h*|x)，即最小化负对数似然 Lsft(θ)=E[−log pθ(h*|x)]，使 LLM 熟悉 hint 格式与生成逻辑。

2.4 强化微调（RFT）。 PPO 依赖价值模型（额外内存/算力）；DPO 省去价值模型但需成对偏好数据（标注复杂，且可能降低生成偏好的绝对概率）。SEFRQO 采用 GRPO——让同一 prompt 生成多个输出，用自定义奖励函数算奖励并做组内优势估计 Â，无需价值模型也无需标注数据。

2.5 基于嵌入相似度的检索。 RAG 通过从向量库检索外部知识来增强 LLM，不改变模型权重。每个参考查询 di 经编码器 φ(·) 映射为 m 维嵌入 ei，组成矩阵 E。对查询 u 计算嵌入 eu，用余弦相似度 sim(u,di)=euᵀei/(‖eu‖‖ei‖) 排序取 top-k 构成检索集 R。

3. 系统概览（System Overview）

3.1 问题定义。 最终目标用 LLM 生成的 hint 降低查询执行延迟，分两阶段：离线微调（让 LLM 会生成 hint）+ 在线上下文学习（让 LLM 生成最优 hint）。定义 1（离线）：最大化 E[log pcorrect(h)]（语法正确）且最小化 E[Latency(Q,h)]（延迟偏好）。定义 2（在线）：对固定参数 LLM，在运行时构造优化 prompt P* 使 h←LLM(P*) 最小化该查询延迟，即 argminP Latency(Q,h)。

3.2 SEFRQO 概览。 Figure 1 展示三阶段，分别对应公式 5/6/7：① SFT（对应公式 5）从标注数据学 hint 格式，奠定系统基础；② QGRPO（对应公式 6）用执行引擎的真实延迟反馈把偏好导向低延迟 hint，作为动态环境中的 RL 过程增强未见 workload 的推理能力；③ 在线 RAG 提示优化（对应公式 7），维护持续更新的 RAG（含实时执行记录），对每个 Q 动态构造含「相似查询参考 + 本查询历史执行分析」的 prompt，重复遇到 Q 时持续优化——自进化过程进一步最小化延迟。

5. Query Group Relative Policy Optimization

该阶段通过把模型输出导向低延迟 hint 来进一步增强 LLM 的查询优化能力。QGRPO 基于 GRPO：对同一 prompt 生成多个 hint {h₁…h_G}，用梯度下降依据这些 hint 的组内优势（对应相对延迟改进奖励）更新参数，优化为 θGRPO。Figure 3 展示工作流与基于延迟的奖励范式。

5.1 基于延迟的奖励模型（Algorithm 2）。 先执行不带 hint 的查询得基线延迟 td（行 1）；对每个 hint hi 执行得实际延迟 ti（行 3）；算改进比 Δti=td/ti（行 4，越大越好）；奖励 ri=RLatency(Δti)（行 5），其中 RLatency(Δti)=tanh(ln(Δti))（公式 8）。ln 带来对称奖励（加速 2 倍 ln2≈0.69，变慢 2 倍 ln0.5≈−0.69），tanh 把奖励限制在 [−1,1]，降低极端偏差的敏感度、提升稳定性。若 ti<td 则 Δti>1、奖励为正，反之负。

5.2 组内相对优势反馈（公式 9）。 Ai=(ri−mean(r))/std(r)。计算新策略与旧策略生成 hi 的概率比 πθ(hi|x)/πθold(hi|x) 指导更新方向。简化梯度 ĜGRPO=Âi+β·(πref(hi|x)/πθ(hi|x)−1)（公式 10）。Algorithm 3 给出完整流程：初始化 πθ=πref=πsft，每步保存旧策略、采样 G 个 hint、算奖励、算优势、用公式 10 更新。

6. 基于 RAG 的提示优化（RAG-based Prompt Optimization）

离线微调后，系统需在在线阶段持续进化以生成更低延迟 hint。对固定微调 LLM，同一 Q 的不同 prompt 导致不同 hint 质量，进而影响延迟（公式 7：优化 prompt x→x* 使 LLM 生成最小化延迟的 h）。本节提出基于 RAG 的 prompt 优化方法，用历史执行记录动态构造 prompt，实现在线上下文学习的自进化。

6.1 自进化 RAG 架构（Figure 4）。 三阶段：Preparation 收集过往执行记录（含 PostgreSQL 默认优化器计划），无需预定义 workload，全部装入 VecDB 引导系统；Initialization 首次遇到 Q 时直接执行（不加 hint）并把记录存 VecDB；Generation 后续遇到 Q 时，先发往执行引擎抽取统计 stats（不实际执行），并行发往 VecDB 做相似度检索得 k 条参考，LLM 据此生成 hint，执行后若延迟低于历史记录则替换 VecDB 中旧记录；再次遇到 Q 时检索其当前最佳历史记录算反馈信息（如相对默认计划的性能增益 η）并入动态 prompt，驱动持续自进化。

6.2 向量库存储设计（Figure 5）。 每条记录维护 7 个属性：Id（主键）、Iteration（Preparation/Initialization 为 0，Generation 从 1 起）、Vector（SQL 文本的嵌入，用于相似度）、SQL_id（基准名+查询名如 job_33b，确保只检索相似且不同的查询）、SQL、Plan、Execution Time。新记录按其布局插入；查询 Q 经编码器（如 SentenceTransformer）编码为定长向量，按余弦相似度取 top-k 参考用于构造 prompt。Figure 5 给出了存储示例（Id=0 执行 300ms、Id=7 第 4 轮 job_33b 执行 1234ms 等）。

📌 对应图 6（Figure 6: Dynamic prompt in SEFRQO）：prompt x=(SP∥UP)。SP 静态（定义角色、动作、约束）；UP 动态，含 k 对相似查询-答案、查询及其统计、默认执行计划、历史最佳计划与性能增益、对 LLM 的期望（生成更好的、不重复、格式正确）。

6.3 动态 Prompt 构造。 Figure 6 展示 x=(SP∥UP) 完整结构。SP 不变，定义角色（数据库专家）、动作、动作约束。UP 指定 LLM 角色与动作。对查询 Q，动态 UP 初版含 k 个最相似查询及其执行计划作参考（提供强相关上下文），再追加查询统计 stats（表基数、过滤选择性）。其后是当前生成的最佳计划与相对基线的增益 η（启发 LLM 理解当前 hint 状态、瞄准更优 hint），结尾给出期望与约束（生成更好 hint、避免照抄执行引擎或自己历史 hint、保证格式正确）。

6.4 自进化反馈范式。 Figure 4 中部展示该范式：LLM 从同一 Q 的历史执行记录学习以增强当前迭代生成的 hint。例子：JOB 20a 查询，Preparation 阶段 PostgreSQL 默认计划为 NestedLoop(k mk)→NestedLoop(k mk cc)；Generation 第一轮 SEFRQO 生成 NestedLoop(mk k)→NestedLoop(mk k t)；后续迭代进化为 HashJoin(mk k)→NestedLoop(mk k cc)。SEFRQO 悟出两点：① 先 join mk 和 k 有益、用 HashJoin 比 NestedLoop 高效；② NestedLoop(mk k t) 高效应保留。Algorithm 4 给出跨迭代 prompt 优化：取上一轮 hint h 执行得 (plan,t)，取历史最佳 (plan*,t*)，若 t<t* 则更新；取初始化阶段存的基线 t₀，算 η=(t₀−t*)/t₀（η<0 表示比基线差），把 (plan*,η) 并入 UP，拼回 SP 得优化 prompt P*，防止 LLM 重复踩坑、持续催生最优 hint h*。

7. 评估（Evaluation）

7.1 实验设置。 服务器 AMD Ryzen 9700X / 128GB / A6000(48GB)。PostgreSQL 16.4 + pg_hint_plan，MySQL 8.4 测跨 DBMS；Milvus 作向量库；6 个本地 LLM：LLaMA3.1-3B(L3B)/8B(L8B) 及其 SFT 版(LS3B/LS8B)、SFT+QGRPO 版(LSG3B/LSG8B)；SentenceTransformer(Paraphrase-MiniLM-L6-v2) 作嵌入；k=1。变体：默认（full-plan + 向量库参考）、JO（仅 Leading）、NR（仅本查询历史、无 RAG 参考）。基线：Bao、Balsa、LLMSteer（LLM-QO/LLMOpt 无完整代码未能复现）。指标：性能增益（Overall/Filtered，↑）、相对执行时间 RET（↓）、推理与 RAG 检索延迟、同质性比率 HR（↓，无效或照抄默认计划的占比）。Benchmark：JOB（微调+预填 RAG）、CEB、Stack、TPC-DS(SF=4)；超时阈值 JOB/CEB 10s、Stack 30s、TPC-DS 60s。

7.2 查询执行延迟评估。 静态场景：CEB（同 schema 不同分布）、Stack（完全未见过的跨域）；动态场景：CEB/JOB/Stack 混合轮流，10 轮 episode 共 50 迭代。Table 1 显示 CEB 上 LS8B 最佳（60.04% Overall），且 SFT 至关重要（L3B/L8B 远差于 LS3B/LS8B）；有趣的是 QGRPO 在 SFT 后略降（因 JOB↔CEB 相似，SFT 知识被充分利用），但在未见 Stack 或关掉 RAG 时，SFT+QGRPO 的探索性优势显现。Stack 上 LS8B/LSG8B 的 Filtered 增益可比拟甚至超过在其上训练的 LQO，证明知识迁移能力。Figure 7 显示 SEFRQO 在所有百分位都优于 PostgreSQL，CEB 上长尾查询约 2.5× 加速。与 LLMSteer 比：CEB 上 LLMSteer Overall 45.14%、SEFRQO 60.04%。动态场景 Table 2：SEFRQO 大幅优于 Bao（14.82%），证明摆脱特定 workload 约束的能力。

📌 对应图 8 / 图 9：CEB / Stack 上随迭代的 RET 曲线（↓）。LS8B 在 CEB 达最低 RET 0.41 且随迭代持续下降；L3B/L8B 原始模型几乎无提升（L3B 恒定生成同一 hint）。

7.3 自进化反馈范式。 Figure 8 显示 CEB 上 LS8B 最佳（RET 0.41）且持续下降潜力；原始模型（L3B/L8B）几乎无增益（L3B 恒定生成同一 hint，未能利用动态 prompt）。领域微调至关重要：LS3B/LS8B 均优于原始版，但 LS3B 比 LS8B 更不稳定。与 LQO 比：Balsa 每轮内改进有限，其好查询散落在不同迭代，需更多迭代才达峰值；Figure 9 中 Stack 上 Balsa 在迭代范围内未能超过基线。Stack 上 LLM 演化更稳（因未微调于 Stack，更谨慎），使小模型能超越大模型——LSG3B（含 QGRPO）在 Stack 最佳。

7.4 SFT 与 QGRPO 效果。 去掉 RAG 自进化（动态 prompt 仅含 PostgreSQL 记录），Figure 10 显示原始 LLM 主要复读默认计划或生成无效 hint（RET≈1、HR≈1），微调 LLM 产生更多样计划（更低 HR）、 favor 探索。3B 模型上 QGRPO 效果显著：LS3B→LSG3B 的 HR 与 RET 进一步降低，LSG3B 甚至优于更大的 LS8B——成功的 QGRPO 能把输出偏好导向比无 QGRPO 的大模型更好的任务性能。但 QGRPO 的 RL 特性带来不确定性，LSG8B 未优于 LS8B（可能 8B 推理更复杂）。

7.5 推理与 RAG 检索时间。 LS3B 推理约 236ms(A6000)/160ms(A100)，RAG 检索平均 0.7ms；主要开销来自 LLM 推理（可用强 GPU 缓解）。CEB 上 SEFRQO 计划平均 520.67ms vs PostgreSQL 922.86ms，端到端仍更低。Figure 11 展示两类典型错误：重复文本块（Repeated Text Chunks，表名重复）、括号不匹配（Bracket Mismatching）——建议推理时加校验，生成不可用 hint 立即中止以降低延迟。

7.6 消融实验。 Figure 12a 显示 LSG3B-JO 的 HR 约 70%（远高于其他），Table 3 显示 full-plan 模式（50.13%）远优于 JO 模式（24.81%）。对比 LSG3B 与 LSG3B-NR（Table 3），前者持续更优，证明参考查询设计有效。Table 4：k=1/2/3 的 CEB Overall 增益 53.93%/57.75%/57.99%，HR 0.26/0.20/0.14——k 增大使 hint 更多样，但增益非单调，太少缺信息、太多迷惑 LLM。

7.7 额外研究。 复杂查询：用 GPT-o3-mini 测 TPC-DS 取得 13.28% Overall / 27.81% Filtered 增益，99 条查询中 28 条用了新计划，模板 10 查询加速达 90.63%（Table 5）。基数不准：随机扰动后 LSG8B 仍达 51.50% Overall（轻微下降）。数据漂移：原始 IMDB 与「2000 年后」缩减库每 5 轮切换，出现尖峰但 LS8B 偶尔取得最低延迟。相似度方法：默认余弦比内积/L2 平均高约 3% 增益，内积产生最多样计划（多 150 个）。跨 DBMS：LSG8B 在 MySQL(CEB) 用简单 hint 翻译器获 26.28% Overall 增益。

8. 相关工作（Related Work）

非 LLM 的 LQO。 按生成范式分 steering 与 generative 两类。Steering 类（Bao、LEON、LOGER、Lero）用神经网络精修传统优化器产出的计划，优点是侵入小、保留核心逻辑、减少差计划风险，但受限于所依赖的传统优化器能力。Generative 类（Balsa、Neo、Lemo）直接控制计划生成，可探索更广计划空间，但灵活度带来更高不稳定性与潜在波动。

LLM 与 RAG 用于数据库。 LLM 已用于 knob tuning（DBBert、GPTuner）、Text-to-SQL、代码生成、schema 理解等。RAG 广泛用于文本生成、视觉问答、代码合成，近期也用于数据库领域：SQL 生成检索相似查询与 schema；查询重写（Rafe）检索重写样例；Text-to-SQL（ReFSQL）检索问题-SQL 对降幻觉。近期三个早期尝试 LLMSteer（仅用嵌入模型选 hint 集，轻量但改进有限）、LLM-QO（主要用微调，DPO 需构造成对偏好）、LLMOpt（用 LLM 替换 Bao 流程中组件做计划候选生成与选择）。SEFRQO 采取 LLM-native 视角，通过模型微调与 prompt 优化定义优化问题，其自进化 RAG 具备先前方法难以达到的持续学习能力（SERAG 是其初步工作）。

9. 结论与未来工作（Conclusion and Future Work）

本文提出 SEFRQO——一个用于查询优化的自进化 RAG 系统。它利用 LLM 微调策略避免冷启动、增强跨 workload 泛化；提出结合离线 SFT、QGRPO 与在线上下文学习的两阶段框架；动态生成 prompt 并通过执行反馈持续进化。我们探索了基于 LLM 的查询优化的知识迁移能力，并展示了实际部署潜力。大量实验证明 SEFRQO 优于先前的 LQO 与基线执行引擎，且据我们所知是首个把 RAG 用于数据库查询优化的工作，拓展了该研究领域的边界。当前 SEFRQO 面向 OLAP 查询，因其开销对 OLTP 仍是挑战，留作未来工作。
